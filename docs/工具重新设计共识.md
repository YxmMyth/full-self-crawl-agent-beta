## 一、核心结论

### 1.1 工具粒度原则

- **工具数量不是硬约束**——原 10 个工具的限制可以突破
- **一工具一能力**——不做 enum 多路复用（证据：SWE-Agent 消融实验，细粒度工具优于粗粒度）
- **活跃工具数 8-15 是甜区**——超过 20 个性能下降（Berkeley BFCL）；可通过延迟加载扩展总量（Claude Code 模式）

证据来源：
- Claude Code：22+ 内置工具，通过 ToolSearch 延迟加载，单次活跃 ~8-12
- SWE-Agent（Princeton 2024）：`search_dir` + `search_file` + `find_file` 优于单一 `search(type=...)`
- Google ADK：统一工具 + enum 比独立工具减 41.2% token，但准确率下降

### 1.2 Browse 需要拆分

**现状问题：** 当前 browse 把导航 + 页面渲染 + 硬编码框架检测 + 网络请求捕获 + 结构化摘要 + WM 记账捆绑在一个工具里。Agent 没有灵活选择感知方式的能力。

**Claude Code 对比：** 7 个独立感知工具（navigate / read_page / get_page_text / read_network_requests / read_console_messages / javascript_tool / find），agent 按需选择。

**拆分依据：** Playwright API 存在三个互不可替代的能力层（见 §二）。

### 1.3 Extract → browser_eval，定位拓宽

**现状问题：** 当前 extract 被设计为"执行 JS 提取数据并保存文件"的专用工具（固定返回 record_count / fields / quality），太窄。

**新定位：** 浏览器内通用 JS 执行器。与 bash 是一对——浏览器内 vs 浏览器外。

用途覆盖：
- 探测数据格局：`Object.keys(window).filter(...)`
- 提取嵌入数据：`JSON.parse(document.getElementById('__NEXT_DATA__').textContent)`
- 检查页面状态：`document.querySelectorAll('.item').length`
- 调用页面内 API：`await fetch('/api/v1/pens').then(r => r.json())`

### 1.4 去除硬编码检测

- browse 里的 `__NEXT_DATA__` / `__NUXT__` / `__APOLLO_STATE__` 等硬编码框架检测**违反架构共识§九硬约束第六条："智能优先，信息的组织、筛选、呈现交给 LLM，不硬编码"**
- 行业验证：6 个系统（Browser-Use / Stagehand / AgentQL / Skyvern / Manus / OpenHands）无一有硬编码检测
- 数据探测能力交给 browser_eval，agent 自主决定探什么

### 1.5 WM 记录架构：单例录制 Agent（已达成共识）

Agent 不应因记录而分心。所有 WM Observation 写入由独立的**单例录制 Agent** 负责，与执行 Agent **实时并行**。

**单例 + Producer-Consumer 模式**——所有 session 共享同一个录制 Agent：

```
Session A 的 transcript 增量 ─┐
Session B 的 transcript 增量 ─┼→ [单例 Recording Agent] ─→ observations DB
Session C 的 transcript 增量 ─┘
                                   │
                                   └──→ 执行 Agent 调 read_world_model 查回

Session 结束 → transcript 程序化保存到 artifacts/{domain}/transcripts/（JSONL）
            → maintain_model LLM 函数自动触发（更新 Semantic/Procedural Model）
```

**为什么单例：**
- 多 session 并发时，多个录制 Agent 写 observations 会重叠/冲突
- 单例有全局视角，跨 session 的 location 重叠能正确去重/合并
- 批处理/过滤优化：只有产生新知识的 tool call（browse/bash/browser_eval）才触发处理

**录制 Agent 架构：与执行 Agent 同构（持久对话 + tool-use 循环）**

| | 执行 Agent | 录制 Agent |
|---|-----------|-----------|
| System prompt | 侦察方法论 | 观察纪律 + 记录策略 |
| User input | Briefing | 执行 transcript 增量（程序注入） |
| Tools | 12 个（浏览器 + bash + 认知） | 4 个（read/create/edit/delete observations） |

**录制 Agent 工具集（参考 Claude Code 的 Read/Edit/Write 模式）：**
- `read_observations(location?)` — 查看当前 observations
- `create_observation(location, raw)` — 创建新 observation
- `edit_observation(id, raw)` — 更新已有 observation
- `delete_observation(id)` — 删除冗余/已合并的 observation

**关键设计：**
- 两条 LLM 调用链完全隔离（不同 system prompt，不同 tools），不存在混淆
- 程序定期注入 transcript 增量，不阻塞执行
- **先 read 再改**——跟 Claude Code 编辑代码一样，避免 ID 错位
- **mid-session 知识回忆**——执行 Agent 后期可通过 read_world_model 查回早期发现
- **Observations 可维护**——避免数量爆炸，保持知识层精炼
- 录制 Agent 的输入包含 agent 的**推理文本**——比较、推断、发现关系的过程也是观察
- 可用更轻模型（任务：观察 + 维护结构化知识）
- **Transcript 是不可变审计追踪**——程序化保存（零 LLM 成本），Observations 的任何修改都可追溯

**两种记忆的协作：** 语义记忆存 DB Model（有界，Planner 维护），精准记忆存文件（scripts/samples/，无界，执行 Agent 产生）。Model 是索引，文件是内容。详见架构共识文档 §7.3。

## 二、浏览器侧工具分层方案

### 2.1 三个不可互替的能力层

| 层 | 能力 | page.evaluate JS 能做？ | Playwright API |
|---|------|----------------------|----------------|
| **页面内容层** | DOM 结构、accessibility tree、可见文本、交互元素 | 部分能（DOM），但 accessibility tree 不行 | locator.aria_snapshot()、inner_text()、content() |
| **JS 执行层** | 全局变量、嵌入 JSON、框架状态、DOM 数据提取 | 就是它自己 | page.evaluate() |
| **网络层** | 请求/响应 body、HttpOnly cookies、完整 headers、请求时序 | **不能** | response.body()、context.cookies()、request.all_headers() |

**网络层的不可替代性是关键发现**——HttpOnly cookies、已完成请求的 response body、完整 request headers，这些只有 Playwright API 能获取，JS 执行无法触及。对侦察系统极其重要（API 返回的 JSON 数据、session cookies）。

### 2.2 工具方案（12 个）

工具粒度原则：**每次调用 = 一个清晰的决策**。按参数模式分组，不同参数签名的操作拆为独立工具。

```
浏览器感知（3 个，三个不可互替的能力层）：
  browse(url?, new_tab?, tab?, visual?) — 页面内容快照 + 多标签页导航
                                          导航（如有 url）+ SPA settle 等待
                                          Markdown+HTML 混合格式 + Data Signals + Network 摘要
                                          new_tab=true 开新标签页，tab=N 切换标签页
                                          visual=true 触发截图 + SoM 标注 + vision LLM 描述
                                          回答："页面长什么样，有什么内容和元素"

  read_network(filter?, clear?)        — 网络层信息（借鉴 Claude Code read_network_requests）
                                          filter: string contains 过滤（如 "/api/"、"graphql"）
                                          clear: 读完清空缓冲区，下次只看新请求（默认 false）
                                          返回：请求列表 + response body preview（每条 1000 chars）
                                                POST 额外显示 request body
                                                cookies（含 HttpOnly，只显示 name+flags+domain）
                                          末尾 tip 引导 bash curl 重放
                                          回答："页面背后有哪些 API 调用，返回什么数据，session 是什么"

  browser_eval(script, save_as?)       — 浏览器内 JS 执行（借鉴 Claude Code javascript_tool + CodeAct）
                                          script: JS 代码（page context，支持 async/await，不需要 return）
                                          save_as 可选：保存到 artifacts/{domain}/samples/{save_as}
                                          在当前活跃标签页执行（不需要 tabId）
                                          返回标注 type + size；大结果 >50KB 自动落盘 workspace/
                                          错误返回 raw error + 程序化 hints（常见错误模式匹配）
                                          超时 30s
                                          回答："页面 JS 上下文里有什么数据，拿出来"

浏览器管理（1 个）：
  browser_reset(proxy?, browser_type?, headed?)
                                       — 重启浏览器到新配置
                                          所有参数可选，裸调用 = 干净重启（同配置）
                                          每个 agent 独占一个浏览器实例，重启不影响其他 agent
                                          自救场景：被检测换代理、切 Chromium、切有头模式调试

页面交互（5 个，原 interact 按参数模式拆分）：
  click(target)                        — 点击元素
                                          target 为元素编号
                                          内置智能：自动识别 <select> → 转下拉选项展示

  input(target, value)                 — 输入/选择
                                          自动识别元素类型（text input → fill，select → 选择选项）
                                          内置智能：autocomplete 字段自动延迟等待建议
                                          内置智能：执行后验证实际值，不匹配时警告 agent

  press_key(key, target?)              — 按键
                                          支持组合键格式："Enter"、"Escape"、"Ctrl+A"
                                          target 可选：指定元素上按键，省略则全局

  scroll(direction?, amount?, target?) — 滚动
                                          direction: up/down/left/right（默认 down）
                                          amount: 屏数（默认 1）
                                          target 可选：指定容器元素内滚动

  go_back()                            — 浏览器后退

系统执行（1 个）：
  bash(command, timeout?)              — 浏览器外代码执行（借鉴 Claude Code BashTool）
                                          command: Shell 命令
                                          timeout 可选：超时毫秒（默认 120000，最大 600000）
                                          每次 spawn 新进程（无状态），工作目录固定 workspace/
                                          输出上限 30,000 chars，尾部截断（同 Claude Code）
                                          大输出自动落盘 workspace/ + preview + 文件路径
                                          返回始终包含 exit code
                                          API 重放、数据处理、文件操作、HTTP 请求

认知辅助（2 个）：
  think(thought)                       — 无副作用推理
  read_world_model(location?)          — 查询 World Model
                                          无参数：返回完整 Semantic + Procedural Model
                                          有 location：返回该 location 的 Observations
                                          两层检索：Model 是索引，Observations 是证据
```

### 2.2b Research Subagent 工具集（独立于执行 Agent）

**Research Subagent** 不是执行 Agent 的工具，是 Planner 按需 spawn 的独立 agent（详见架构共识文档 §7.4b）。它有自己的工具集：

```
web_search(query, domain?, save_as?)   — 搜索引擎查询
                                          domain 可选：限定在某个域名内搜索（site: 语法）
                                          save_as 可选：大结果保存到 research/
                                          后端可切换：DuckDuckGo（MVP）/ Brave / Tavily

web_fetch(url, save_as?)               — 轻量 HTTP 获取
                                          不走浏览器，快速获取纯文本/JSON/API 文档
                                          跟执行 Agent 的 browse 区分：browse 走浏览器渲染，web_fetch 是纯 HTTP

bash(command)                          — 兜底能力（curl / gh api / 数据处理）

think(thought)                         — 无副作用推理
```

共 4 个工具。Research Subagent 的输出是调研报告 markdown 文件，写入 `artifacts/{domain}/research/{topic}.md`。

### 2.2c Verification Subagent 工具集（DONE 守门人，feature-gated 可拆卸）

**Verification Subagent** 是 `mark_done` 工具的 Python 实现内部自动触发的反 satisficing 子 agent（详见架构共识文档 §7.4c）。Planner 不直接 spawn 它，不需要知道它存在。通过 `VERIFICATION_SUBAGENT_ENABLED` 环境变量控制，可完全拆卸——禁用后 mark_done 直接返回 DONE。

它的工具集极简——**只读 + 执行**，不能写 WM：

```
read_world_model(location?)            — 读 World Model（跟执行 Agent 同一个工具）
                                          无参数：返回完整 Semantic + Procedural Model
                                          有 location：返回该 location 的 Observations
                                          严格只读，不能 create/edit/delete observations

bash(command)                          — 执行验证命令
                                          curl / curl_cffi 验 API 可达性和响应
                                          读 artifacts/{domain}/samples/ 验样本存在性和格式
                                          跑 artifacts/{domain}/scripts/ 里的脚本验证方法有效性
                                          数据处理 / 对比 / 统计

think(thought)                         — 无副作用推理
```

共 **3 个工具**（比 Research Subagent 的 4 个还精简）。Verification Subagent 的输出是验证报告 markdown 文件，写入 `artifacts/{domain}/verification/round_N.md`，最后一行必须是 `VERDICT: PASS` / `VERDICT: FAIL` / `VERDICT: PARTIAL`（被 Planner 程序解析）。

**工具选择的权限约束：**

| 权限 | 执行 Agent | Research Subagent | Verification Subagent |
|------|-----------|-----------------|---------------------|
| 操作目标站点（浏览器） | ✓ | ✗ | ✗（MVP，未来可加 browser_eval 只读模式） |
| 搜索互联网 | ✗ | ✓ | ✗ |
| 读 WM | ✓ | ✗ | ✓ |
| **写 WM（通过录制 Agent 间接）** | ✓ | ✗ | ✗ |
| 执行 bash | ✓ | ✓ | ✓ |

**设计理由：**
- **严格只读**——Verification Subagent 不能修改 WM，避免它"自己创造符合自己判断的观察"
- **不给浏览器工具**——MVP 最小化，bash 通过 curl 足以验证 API、文件、脚本
- **给 bash**——对抗 satisficing 的关键是"必须跑命令"，没有 bash 就只能读，变成纸上谈兵

### 2.2d Planner 工具集（tool-use agent）

**Planner** 是战略决策层，以 tool-use agent 形式运行（详见 `docs/Planner设计.md`）。它有 5 个工具：

```
spawn_execution(briefing)              — 派发一次执行 session
                                          内部 pipeline：Session → maintain_model → 返回 summary
                                          返回：{ summary, model_diff, new_obs_count, session_id }

spawn_research(topic, questions)       — 派发一次互联网调研
                                          启动 Research Subagent（4 个工具）
                                          返回：{ key_findings, report_path }

read_model()                           — 查看当前 Semantic + Procedural Model
                                          幂等，永远返回 DB 最新版本
                                          返回：{ semantic_model, procedural_model, version }

think(thought)                         — 无副作用推理

mark_done(reason)                      — 标记完成
                                          内部自动触发 Verification（如果 feature 启用）
                                          成功返回：{ status: "DONE" }
                                          被驳回返回：{ status: "blocked", verdict, gaps }
```

共 **5 个工具**。Planner 不直接操作浏览器、不直接维护 observations、不直接更新 Model。

**工具选择的权限约束（完整对比）：**

| 权限 | Planner | 执行 Agent | Research | Verification |
|------|---------|-----------|----------|-------------|
| 派发子 agent | ✓ | ✗ | ✗ | ✗ |
| 操作目标站点（浏览器） | ✗ | ✓ | ✗ | ✗ |
| 搜索互联网 | ✗（通过 spawn_research） | ✗ | ✓ | ✗ |
| 读 Model | ✓ | ✓（read_world_model） | ✗ | ✓ |
| 写 WM | ✗ | ✓（通过录制 Agent 间接） | ✗ | ✗ |
| 更新 Model | ✗（由 maintain_model 自动） | ✗ | ✗ | ✗ |
| 执行 bash | ✗ | ✓ | ✓ | ✓ |

### 2.3 内置智能行为（基础设施层）

以下行为在工具实现内部自动执行，不增加工具数量。参考 Browser-Use 的设计：

| 行为 | 触发时机 | 效果 |
|------|---------|------|
| **click 元素类型识别** | click 目标是 `<select>` | 自动转为展示下拉选项，agent 不需要区分 click vs select |
| **input autocomplete 检测** | input 目标有 `combobox`/`aria-autocomplete` 属性 | 自动延迟 400ms 等待建议列表 |
| **input 值验证** | input 执行后 | 检查字段实际值是否匹配，被格式化/自动修改时警告 |
| **新元素标记 `*`** | 任何交互后的页面快照 | 新出现的元素标 `*` 前缀，agent 一眼看出什么变了 |
| **页面变化保护** | 交互导致 URL 变化 | 自动更新 World Model 记录导航，返回新页面快照 |
| **对话框自动处理** | alert/confirm/prompt 弹出 | 自动 accept（侦察场景中弹窗是干扰） |

### 2.4 设计决策记录

- **interact 拆分为 5 个工具**——按参数模式分组（click 只需 target；input 需 target+value；scroll 需 direction）。证据：SWE-Agent 细粒度工具优于粗粒度；Claude Code 也拆 Read/Edit/Write
- **note_insight 不再需要**——录制 Agent 全权负责 Observation 写入（见 §1.5）
- **标签页管理集成到 browse**——agent 的决策是"看什么"，不是"管理标签页"
- **browser_reset 是一个工具而非多个**——"重启并配置"是一个完整决策
- **每个 agent 独占浏览器实例**（1 agent = 1 site = 1 browser），避免共享冲突
- **内置智能行为不暴露为工具**——放在基础设施层，让每个工具更聪明而不是更多工具

### 2.3 行业空白：数据探测

调研 6 个系统，**没有任何系统有"主动发现页面数据源"的能力**。所有系统都靠 agent 从页面内容中自行推断。

我们是侦察系统，数据探测恰恰是核心场景。browser_eval 天然支持通用数据探测，无需硬编码：
- 扫描全局变量：`Object.keys(window).filter(k => typeof window[k] === 'object' && JSON.stringify(window[k]).length > 1000)`
- 扫描 script 标签：`[...document.querySelectorAll('script')].map(s => ({type: s.type, size: s.textContent.length}))`
- 检查框架状态：由 agent 自行决定查什么，不预设检测列表

### Playwright 能力调研
- Playwright 1.58.0 完整 API 审查（结构层 / 数据层 / 网络层）
- 关键发现：网络层能力不可由 page.evaluate 替代

### 行业系统对比（6 个）
- Browser-Use：感知自动隐式，提取 1 个专用工具
- Stagehand v3：observe() + extract() 最清晰分离
- AgentQL：query_elements vs query_data 按"元素 vs 数据"分
- Skyvern 2.0：三个独立 agent（感知 / 交互 / 提取）
- Manus：browser_view 自动感知，无专用提取（JS + file_write 组合）
- OpenHands：AX tree 自动感知，无专用提取（code-as-extraction）

### 学术证据
- SWE-Agent（Princeton 2024）：~10 工具甜区，细粒度优于粗粒度
- Berkeley BFCL：>20 候选工具时准确率显著下降
- D2Snap（arXiv:2508.04412）：Markdown+HTML 混合 73% 最高准确率
