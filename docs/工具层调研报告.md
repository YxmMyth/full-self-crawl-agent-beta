## 零、跨系统宏观：工具设计哲学

> 本节调研 8 个生产级 agent 系统的工具设计，提炼跨系统的共性模式。

### 0.1 各系统工具概况

| 系统 | 工具数 | 粒度 | 核心设计选择 | 来源 |
|------|--------|------|------------|------|
| Claude Code | ~18 | 每工具一个认知功能 | 语义化专用工具，超详细描述，结构化防错（Edit 要求先 Read） | 系统 prompt 逆向 + Anthropic 工程文档 |
| Anthropic CUA | 3 | 像素级原语 | Schema 内嵌模型权重，不可定制 | Anthropic 官方文档 |
| OpenAI CUA | 1 | 单工具+双模式（GUI/代码） | 动作批处理，Playwright 代码模式作为逃生通道 | OpenAI 官方文档 |
| Manus | ~21 | 前缀分类的原子动作 | KV-cache 优先，logit masking 控制可用子集，文件系统当外部记忆 | 泄露 prompt + Context Engineering 博文 |
| Devin | ~35 | 原子+LSP 集成 | 并行动作发射，语义搜索替代 grep | 泄露 prompt + 官方文档 |
| Browser-Use | ~20 | 原子+LLM 提取 | 结构化输出（非 tool calling），索引定位，domain 过滤 | 源码分析 |
| Skyvern | 21 | 两层（自然语言→原子） | 视觉优先，planner-actor-validator 三阶段 | 源码 + 博客 |
| OpenHands | 6-9 | 代码即动作 | 浏览器包装为单工具内 DSL，最少工具集 | ICLR 2025 论文 + 源码 |

### 0.2 工具数量与性能

量化证据表明工具选择准确率随数量非线性退化：

- 4 工具 ~1,200 tokens → ~95% 准确率；46 工具 ~42,000 tokens → ~71%（多方汇总）
- 从 5 扩展到 200 个 skill，选择准确率存在**相变点**，根因是语义混淆而非数量本身（arXiv:2601.04748）
- OpenAI 指导：<100 工具 + <20 参数/工具 为安全区
- ToolLLM (ICLR 2024)：层级路由 66.7% vs 扁平 ReACT 47.0%（+42%）
- RAG-MCP：检索式工具选择将准确率从 13.6% 提升至 43.1%，同时 prompt tokens 减半
- 实践甜区：**10-25 个工具**，前提是语义可区分

### 0.3 工具描述质量

调研中信号最一致的发现——描述质量是最高杠杆的优化点：

- Anthropic 工程文档：*"Even small refinements to tool descriptions can yield dramatic improvements"*——Claude Sonnet 3.5 靠打磨工具描述拿到 SWE-bench Verified SOTA
- Gorilla (NeurIPS 2024)：更好的 API 文档 → 准确率 +20.43%，部分场景零幻觉
- API-Bank (EMNLP 2023)：模型间最大差距在工具**选择**而非执行，选择精度直接取决于描述质量
- Claude Code 做法：每个工具描述是一篇小文档（数百 token），包含用途、约束、反模式、具体例子、什么时候不该用

Anthropic 建议把工具描述当作"给新人的入职手册"来写。

### 0.4 返回值设计

所有系统趋同：返回压缩/摘要后的观察，完整数据存外部。

- Claude Code：工具返回上限 25,000 tokens，旧结果在 compaction 时清除
- Anthropic 指导：返回高信号内容，避免底层标识符（uuid、mime_type），优先语义标识（name、file_type）
- 支持 detailed/concise 模式切换——Slack 场景 72 vs 206 tokens，功能无损
- 大数据用指针模式：记录数 + 字段名 + 前 N 条预览 + 文件路径（arXiv:2511.22729）
- 子 agent 摘要：1,000-2,000 tokens（Anthropic Context Engineering 文档）

### 0.5 结构化防错（poka-yoke）

Anthropic 提出 ACI（Agent-Computer Interface）概念——工具接口设计应投入等同 HCI 的精力：

- Claude Code：Edit 未 Read 过的文件直接**拒绝执行**（结构化依赖，不是建议）
- 发现 agent 用相对路径总出错 → 改成强制绝对路径 → "the model used this method flawlessly"
- 核心原则：*"Change the arguments so that it is harder to make mistakes"*
- enum 优于自由文本（当值域有界时），必填参数明确化，扁平参数优于深层嵌套

### 0.6 静态工具集

所有系统共识：session 内不动态增删工具。

| 系统 | 做法 |
|------|------|
| Manus | 静态定义 + logit masking 控制可用子集（prompt 不变，保护 KV-cache） |
| Devin | 35 工具全部常驻 |
| Browser-Use | 基本静态，actions 可按域名过滤但定义不变 |
| OpenHands | 初始化时配置，session 内不变 |

Manus 原话：动态增删工具会破坏 KV-cache prefix，对 100:1 输入输出比意味着巨大成本。

### 0.7 指导分工

跨工具的行为策略放系统 prompt，单工具的用法放工具描述：

- Claude Code：系统 prompt 建立�先级层次（"优先 Edit 而非 sed"），工具描述放操作细节
- Manus：系统 prompt 有 `<browser_rules>` / `<coding_rules>` 跨工具策略区块
- Devin：系统 prompt 指定工具偏好（"Never use grep, use semantic_search"）

共性：涉及**多工具选择/优先级**的指导 → 系统 prompt；涉及**单工具参数/场景**的指导 → 工具描述。

### 0.8 浏览器动作粒度

无系统使用"填写表单"等高层复合动作。差异在于一次 LLM 调用中能打包多少原子操作：

| 模式 | 系统 | 特点 |
|------|------|------|
| 单动作/轮 | Manus | 每次必须观察结果再决定下一步 |
| 多动作批处理 | Browser-Use（≤5）, OpenAI CUA | 独立动作可并行，改变页面的动作截断后续 |
| 代码即动作 | OpenHands | 浏览器操作写成 DSL 代码字符串 |
| 并行发射 | Devin | "Output multiple actions at once, as long as they can be executed without seeing the output" |

### 0.9 工具组合：扁平，不嵌套

**没有任何系统允许工具调用工具。** 所有架构都是扁平的：agent 调用工具 → 观察结果 → 决定下一个工具。唯一的"嵌套"形式是 OpenHands 的 agent 委托 agent（AgentDelegateAction），但那是 agent 层面而非工具层面。

## 一、感知：agent 怎么看

**核心事实：56.7% 的 CUA agent 点击打到了错误目标（Visual Confused Deputy, arXiv:2603.14707）。感知精度是整个系统的瓶颈，不是规划能力。**

AgentOccam（ICLR 2025）证明仅优化观察空间的效果比改进 agent 架构高 161%。投资在"agent 能看到什么"上的回报远大于"agent 怎么想"。

### 1.1 页面表示方式——深度对比

#### 可访问性树（AX Tree）

Playwright 提供两种 AX tree 接口：

**`locator.ariaSnapshot()`**（推荐）返回 YAML 格式：
```yaml
- banner:
  - heading "Playwright enables reliable end-to-end testing" [level=1]
  - link "Get started"
  - link "Star microsoft/playwright on GitHub": Star
- main:
  - img "Browsers (Chromium, Firefox, WebKit)"
  - heading "Any browser - Any platform - One API"
```

**`page.accessibility.snapshot()`**（已弃用）返回 JSON 对象，含 `role`、`name`、`children`、`focused` 等属性。默认过滤"无趣"节点（装饰性容器）。

**AX tree 的真实局限：**

| 问题 | 影响 | 来源 |
|------|------|------|
| **iframe 内容不包含** | CDP `Accessibility.getFullAXTree` 不返回 iframe 子节点。iframe 节点角色为 `Iframe` 但 `children` 为空 | Puppeteer issue #7744 |
| **Shadow DOM 关系断裂** | ID 引用（`for`, `aria-describedby`, `aria-activedescendant`）不能跨 shadow boundary。浏览器"展平"shadow tree 暴露内部节点，但语义关系丢失 | Alice Boxhall (Igalia), Nolan Lawson |
| **Closed shadow root 不可访问** | Playwright 的 CSS/text locator 可穿透 open shadow root，但 closed shadow root 是硬限制 | Playwright 文档 |
| **ARIA 标注差的站点** | WebAIM 百万页面调查：有 ARIA 的页面平均比没有 ARIA 的多 41% 检测错误。差的 ARIA 积极误导 | WebAIM 年度报告 |
| **自动化工具只捕获 20-25% 可访问性问题** | 大多数真实站点的 AX tree 是不完整的页面图景 | 多方佐证 |
| **通用容器被过滤** | 无语义角色的 `<div>` 显示为 `genericContainer` 并被 `interesting_only=true` 过滤 | Playwright 文档 |

**来源：** [alice.pages.igalia.com](https://alice.pages.igalia.com/blog/how-shadow-dom-and-accessibility-are-in-conflict/), [nolanlawson.com](https://nolanlawson.com/2022/11/28/shadow-dom-and-accessibility-the-trouble-with-aria/), Puppeteer issue #7744, Playwright 文档。

#### 增强 DOM 树

Browser-Use 的实际处理管线：

1. **JS 注入**（`buildDomTree.js`）：递归遍历 DOM，检查可见性、可交互性、最前面状态
2. **简化**：过滤 script/style/metadata；`ClickableElementDetector` 检测可点击元素
3. **Paint order 过滤**：被高 z-index 元素遮挡的移除
4. **Bounding box 过滤**：完全包含在可点击容器内的子元素合并（99% 包含阈值）
5. **索引分配**：交互元素获得顺序编号

~10,000 原始 DOM 节点 → ~200 个带索引的交互元素。处理时间 10-100ms/页面。

Browser-Use 给 LLM 看的实际格式：
```
[1]<button type="submit" /> Submit Form
[2]<input type="text" placeholder="Enter name" />
|SCROLL|<div class="content" /> (2.1 pages above, 1.3 pages below)
*[3]<a href="/home" /> Home          ← * 标记新出现的元素
```

`|SCROLL|` 标记显示滚动位置上下文。新元素（AJAX/modal 动态加载）前缀 `*`。

**来源：** [deepwiki.com/browser-use](https://deepwiki.com/browser-use/browser-use/2.4-dom-processing-engine), Browser-Use 源码。

#### D2Snap 研究（关键发现）

论文 "Beyond Pixels: Exploring DOM Downsampling for LLM-Based Web Agents"（arXiv:2508.04412）提供了最严格的格式对比：

| 方式 | 平均 Token | 成功率 |
|------|-----------|--------|
| 原始截图 | 2,294 | 55% |
| 标注截图（+ bounding box） | 3,754 | 65% |
| 原始 DOM（截断 8k） | 8,121 | 59% |
| D2Snap 下采样 DOM | 18,943 | **73%** |
| D2Snap 线性化 | 7,178 | 67% |

**核心结论：**
1. **DOM 层级是最重要的特征**——丢弃层级（扁平化提取）一致性地降低性能
2. **内容转 Markdown + 交互元素保留 HTML** 是最佳混合格式
3. **视觉数据的边际收益很低**——标注截图（65%）vs 纯文本标注截图（63%）几乎相同

输出格式示例：
```html
<section class="container" required="true">
# Our Pizza
<button type="button">Add</button>
</section>
```

Cloudflare 的 "Markdown for Agents" 在基础设施层验证了这一点：HTML → Markdown 转换减少 80% token 同时保留语义结构。

**来源：** arXiv:2508.04412, [blog.cloudflare.com/markdown-for-agents/](https://blog.cloudflare.com/markdown-for-agents/)

### 1.2 元素索引系统——深度对比

四个主要系统的详细机制：

#### Browser-Use：backendNodeId + SHA-256 哈希

- 每个交互元素用 CDP `backendNodeId` 标识
- 元素身份通过三部分 SHA-256 哈希：`parent_branch_path | attributes_string | ax_name`
- 属性只保留 ~50 个静态属性（class, id, name, type, role, aria-*, data-test*, href, src 等）
- 动态 CSS class 被过滤（hover, active, selected, disabled, animation, loading 等）生成 `stable_hash`
- `selector_map` 在每次状态请求时重建，不跨导航持久化

**来源：** Browser-Use 源码 `dom/views.py` line 861, `dom/serializer/serializer.py`。

#### Stagehand：LLM 每次重新推理

- `observe()` 分析 AX tree，返回包含 XPath + 描述 + 动作类型的序列化描述符
- `act()` 从自然语言或缓存的 `ObserveResult` 执行
- 缓存 selector 失败时，回退到完整 LLM agent "多维自愈"
- v3 从 Playwright 切换到直接 CDP，复杂 DOM 交互快 44%

**来源：** [stagehand GitHub](https://github.com/browserbase/stagehand), [browserbase.com/blog/stagehand-v3](https://www.browserbase.com/blog/stagehand-v3)

#### Playwright MCP：短暂 ref

格式示例：
```yaml
- generic [ref=e2]:
  - heading "MCP Inspector v0.16.4" [level=1] [ref=e8]
  - combobox "Transport Type" [ref=e13]:
    - option "STDIO" [selected]
    - option "SSE"
  - textbox "URL" [ref=e17]
  - button "Connect" [ref=e21]
```

- 每个交互元素获得 `[ref=eN]` 标识符
- Ref 在每次状态变化 action 后重新生成
- 过期 ref 产生错误：`"Could not find element with ref: button-123"`
- 不尝试跨 action 持久化身份

**来源：** [playwright-mcp GitHub](https://github.com/microsoft/playwright-mcp), DeepWiki 分析。

#### Agent-Browser（Vercel）：@eN ref

```
@e1 [heading] "Example Domain" [level=1]
@e2 [button] "Submit"
@e3 [input type="email"] placeholder="Email"
@e4 [link] "Learn more"
```

iframe 内联并缩进：
```
@e2 [Iframe] "payment-frame"
  @e3 [input] "Card number"
  @e4 [button] "Pay"
```

页面变化后 ref 失效，支持 `-i` flag 只显示交互元素。

**来源：** [agent-browser.dev/snapshots](https://agent-browser.dev/snapshots)

#### 对比总结

| 维度 | Browser-Use | Stagehand | Playwright MCP | Agent-Browser |
|------|------------|-----------|----------------|---------------|
| ID 类型 | backendNodeId + hash | LLM 推理 | 短暂 ref | 顺序编号 @eN |
| 跨页面持久性 | 5 级回退匹配 | LLM 自愈 | 无，重新快照 | 无，重新快照 |
| iframe 支持 | CDP session per frame | DFS 遍历 + 复合 ID | frameLocator | 内联缩进 |
| Shadow DOM | 显式例外（form 元素） | 自动（iframe 内限制） | 仅 open | 未记录 |
| 大页面策略 | 视口过滤 + 列表截断 | AX tree（减少 80-90%） | 无分页 | `-i` 交互过滤 |

### 1.3 网络请求观察

**被动捕获**是感知层的重要补充渠道。SPA 的 DOM 是 API 数据的渲染视图——捕获 API 响应比抓 DOM 更干净、更完整。

Playwright 提供：
- `page.on('request')` / `page.on('response')`：被动监控所有 HTTP/HTTPS 流量
- `page.route()`：拦截和修改
- `page.routeWebSocket()`：WebSocket 拦截
- HAR 录制：`browser.newContext({ recordHar: {...} })`

**呈现格式**（推荐）：
```
API Requests Observed:
- GET /api/v1/pens?tag=threejs&page=1 → 200 (JSON, 15 items)
- GET /api/v2/search?q=webgl → 200 (JSON, 25 items)
- POST /graphql {query: "PensByTag"} → 200 (JSON)
```

**噪声过滤**：
- 按 URL pattern：数据 API 通常含 `/api/`, `/graphql/`, `/v1/`；分析跟踪走 `google-analytics.com`, `segment.io` 等域名
- 按 Content-Type：数据 API 返回 `application/json`；tracking pixel 返回 1x1 GIF
- 按响应大小：数据 API 有实质 payload；tracking 响应通常很小
- 按 HTTP method：数据 API 常用 POST + JSON body；tracking 用 GET + query params

**GraphQL 检测启发式：**
- 请求 URL 含 "graphql"
- POST body 含 `query` 属性 + 合法 GraphQL 语法
- 请求含 `operationName` 字段
- 响应含 `data` 和/或 `errors` key

**"Beyond Browsing" 论文（ACL 2025）量化证据：**

| Agent 类型 | 准确率 |
|-----------|--------|
| 仅浏览 | 14.8% |
| 仅 API/代码 | 29.2% |
| **混合（浏览+API）** | **35.8-38.9%** |

混合 agent 在 77.7% 的任务中同时使用了浏览和 API 两种模态。

**来源：** arXiv:2410.16464, Playwright 文档, Browserless blog。

### 1.4 嵌入数据检测

现代框架将服务端数据直接嵌入 HTML 用于客户端 hydration：

| 模式 | 框架 | 检测方式 | 数据位置 |
|------|------|---------|---------|
| `<script id="__NEXT_DATA__">` | Next.js Pages Router | 固定 ID 的 script 标签 | `props.pageProps` |
| `self.__next_f.push()` | Next.js App Router (RSC) | 内联 script 中的 flight data | 行分隔协议（chunk_id:type:payload） |
| `window.__NUXT__` | Nuxt.js 2/3 | 内联 script | devalue 序列化（支持 Date/Map/Set） |
| `window.__INITIAL_STATE__` | Redux/Vuex | 内联 script | JSON 状态树 |
| `window.__APOLLO_STATE__` | Apollo GraphQL | 内联 script | GraphQL cache |
| `<script type="application/ld+json">` | Schema.org | 标准化标签 | JSON-LD 结构化数据 |
| `<script type="application/json">` | 各种 | 通用 JSON 标签 | 组件数据 |

**`__NEXT_DATA__` 实际结构示例：**
```json
{
  "props": {
    "pageProps": { /* getStaticProps/getServerSideProps 的数据 */ },
    "__N_SSP": true
  },
  "page": "/route/[id]",
  "query": {},
  "buildId": "mgml30LY3PGj-2IRzRxCf"
}
```

真实站点体量：Wellcome Collection 主页有 207KB 的 `__NEXT_DATA__`（占 HTML 总量 50%）。

**Next.js App Router RSC flight data 格式**（更复杂）：
```
<chunk_id>:<type_marker><serialized_payload>
```
类型标记：`I`（导入）、`H`/`HL`（hints）、`D`（数据/组件）、`"`（字符串）、`{`（JSON 对象）。超过 2048 字符的 payload 分割跨多个 `push()` 调用。检测方法：发 GET + `RSC: 1` header，测试响应是否匹配 `/^\d+:["IHL"]/`。

**Python 处理库**：`njsparser`——`get_next_data(html)` 处理 Pages Router，`BeautifulFD(html)` 处理 flight data。

**框架检测脚本：**
```javascript
const framework = {
    react: !!document.querySelector('[data-reactroot]') || !!window.__REACT_DEVTOOLS_GLOBAL_HOOK__,
    vue: !!window.__VUE__ || !!document.querySelector('[data-v-]'),
    angular: !!window.ng || !!document.querySelector('[ng-version]'),
    nextjs: !!document.querySelector('#__NEXT_DATA__') || !!document.querySelector('#__next'),
    nuxt: !!window.__NUXT__ || !!document.querySelector('#__nuxt'),
    svelte: !!document.querySelector('[data-svelte-h]'),
};
```

**来源：** [trickster.dev](https://www.trickster.dev/post/scraping-nextjs-web-sites-in-2025/), [edspencer.net](https://edspencer.net/2024/7/1/decoding-react-server-component-payloads), [scrapfly.io](https://scrapfly.io/blog/posts/how-to-scrape-hidden-web-data), njsparser GitHub。

### 1.5 动态内容感知

**SPA 智能等待——DOM Settle 模式（Stagehand，业界最佳实践）：**
```javascript
async function waitForDomSettle() {
    return new Promise(resolve => {
        let timeout = setTimeout(resolve, 1500); // 1.5s 无变化 = 稳定
        new MutationObserver(() => {
            clearTimeout(timeout);
            timeout = setTimeout(resolve, 1500); // 任何 DOM 变化重置计时
        }).observe(document.body, { childList: true, subtree: true });
    });
}
```

推荐组合方案：`domcontentloaded` → MutationObserver settle（1.5s 静默阈值）→ 特定元素 waitForSelector 确认 → 硬超时兜底（15s）。

**为什么 `networkidle` 不可靠：** SPA 经常有后台 API 调用（分析、轮询、实时更新），网络永远不会完全空闲。Playwright 文档明确不推荐 SPA 使用 `networkidle`。

**实时内容（WebSocket/自动刷新）：** 基本未解决。当内容不经用户交互就变化时，agent 快照立即过期。Playwright 支持 `page.on('websocket')` 和 `page.routeWebSocket()`，但没有 web agent 框架用 WebSocket 监控做感知。

**来源：** Stagehand 源码, Playwright 文档, [browserstack.com/guide/playwright-waituntil](https://www.browserstack.com/guide/playwright-waituntil)。

### 1.6 感知失败模式（量化）

## 二、交互：agent 怎么操作

### 2.1 元素定位——Browser-Use 的 5 级回退链

当需要重新找到一个元素时（页面状态变化后），Browser-Use 依次尝试（`_update_action_indices()`）：

| 级别 | 匹配方式 | 机制 | 鲁棒性 |
|------|---------|------|--------|
| 1. EXACT | 完全哈希匹配 | SHA-256(parent_path \| attrs \| ax_name) | 最精确，任何变化就失败 |
| 2. STABLE | 稳定哈希匹配 | 过滤动态 CSS class（hover/active/selected/loading/animation 等）后哈希 | 抗状态变化 |
| 3. XPATH | XPath 路径匹配 | 字符串比较 XPath（沿 parent chain，含 sibling index） | 抗样式变化 |
| 4. AX_NAME | 可访问性名称匹配 | node type + AX tree 的 accessibility name | 抗结构变化 |
| 5. ATTRIBUTE | 属性匹配 | name, id, aria-label + node type | 最后兜底 |

五级全部失败返回 `None`，action 无法执行。

**来源：** Browser-Use 源码 `agent/service.py` lines 3488-3626, `dom/views.py` line 861。

### 2.2 Stale Reference 问题

**CDP 的三种节点 ID：**

| ID 类型 | 生命周期 | 更新机制 | 开销 |
|---------|---------|---------|------|
| NodeId | 活跃检查模式 | Backend 主动推送变化通知 | 高 |
| BackendNodeId | 页面存活期间 | 无通知——节点销毁时悄悄失效 | 低 |
| RemoteObjectId | JS 执行上下文 | 绑定执行上下文 | 中 |

**关键事实：** BackendNodeId 在页面存活期间有效，但导航后全部失效（CDP 文档确认）。

**业界共识：不要跨页面变化维持持久元素引用。** 所有系统的做法：

1. 任何可能修改 DOM 的 action 后，重新快照页面状态
2. 用语义属性（accessible name, role, 可见文本）而非结构属性（XPath, node ID）重新识别
3. 告诉 LLM 元素索引可能过期，需要重新检查

Browser-Use 的错误消息："If the page changed after navigation/interaction, the index [X] may be stale. Get fresh browser state before retrying."

**来源：** [WICG DevTools Protocol Issue #9](https://github.com/WICG/devtools-protocol/issues/9), CDP DOM 文档。

### 2.3 Action Verification——交互后发生了什么

三种主要方案：

#### Agent-E：MutationObserver 变化观察

- 交互前注入 MutationObserver
- 捕获 childList 变化（新增元素）和 characterData 变化（文本修改）
- 交互后 100ms 取消订阅
- 返回**语言化反馈**："Clicked the element with mmid 25. As a consequence, a popup has appeared with following elements."
- 同时监控 `aria-expanded` 等状态属性

**来源：** [Agent-E paper arXiv:2407.13032](https://arxiv.org/html/2407.13032v1)

#### Browser-Use：新元素 `*` 标记

- 维护 `_previous_cached_selector_map`
- 对比当前快照和上次快照的 `backend_node_id`
- 新元素前缀 `*[12345]<button ...>` vs `[12345]<button ...>`
- 帮助 LLM 注意到"动态内容如 alerts、modals、AJAX 加载的元素"

#### Browser-Use：循环检测器

- 导航超限返回 `"NAVIGATION_LOOP_DETECTED"`
- 同一字段修改超过 2 次无可见变化返回 `"FIELD_INTERACTION_LOOP"`
- `_is_redundant_retry_step()` 检测与上一步完全相同的操作

### 2.4 点击回退链——Browser-Use 的完整实现

Browser-Use 的 `_click_element_node_impl()`（`default_action_watchdog.py` lines 695-1050）：

```
1. DOM.scrollIntoViewIfNeeded(backendNodeId)   ← 滚动到可见
   ↓ (50ms wait)
2. 获取元素坐标（统一方法返回 rect）
   ↓
3. 可见性检查：找到与视口相交的最大可见四边形
   ↓
4. 遮挡检查：document.elementFromPoint(x, y)
   验证目标元素（或语义相关元素如其 label）会接收点击
   ├── 检查双向包含关系
   └── 检查 label-input 关联（向上 3 层祖先）
   ↓
5a. 未被遮挡 → CDP mouse events (mouseMoved, mousePressed, mouseReleased) @ 中心点
5b. 被遮挡 → JS fallback: DOM.resolveNode → Runtime.callFunctionOn('function() { this.click(); }')
5c. 无坐标 → 直接 JS click
   ↓
6. Checkbox 验证：记录 pre-click 状态，验证 post-click 状态变化。未变 → JS fallback
```

**来源：** Browser-Use 源码确认。

### 2.5 复杂交互模式

#### Dropdown 处理

**Native `<select>`：** Browser-Use 有专用 `dropdown_options` 和 `select_dropdown` action。LLM 尝试 `click` 一个 `<select>` 时被拦截："Cannot click on <select> elements. Use dropdown_options(index=X) action instead."

**Custom dropdown：** 通过 ARIA role（`listbox`, `menu`, `combobox`, `menubar`）、tag（`select`）、CSS class（`dropdown`, `dropdown-menu`）检测。依赖 LLM 执行 click-to-open → click-option 序列。

**Invariant Labs 发现：** Dropdown/combobox 失败是四大主要失败类别之一。修复方法：引入 `select_option[id=label]` action 绕过 AX tree 通过标签直接匹配 HTML option。

#### 表单填写

**关键坑：** 输入文字时"字符追加到已有文本而非替换"。修复：fill 前先 `Ctrl+A Ctrl+Backspace` 清空字段。

#### Hover

Playwright 提供 `locator.hover()`。Browser-Use 在点击前 dispatch `mouseMoved` 事件。但没有 agent 框架有显式的 "hover to reveal then interact" 策略——依赖 LLM 推理。

#### 文件上传

Browser-Use 检测 `<input type="file">` 并重定向到 `upload_file_to_element` action。显式处理 Bootstrap 模式的 `opacity:0` 隐藏 file input。

**来源：** Browser-Use 源码, Invariant Labs blog, Playwright 文档。

### 2.6 Frame 和 Shadow DOM 交互

#### iframe 处理

| 系统 | 方案 | 特色 |
|------|------|------|
| Browser-Use | CDP session per frame，cross-origin 通过 URL 匹配回退 | `target_id` + `frame_id` + `frame_path` |
| Stagehand | DFS 遍历，CDP flattened auto-attach，复合 ID（frame ordinal + backendNodeId） | `deepLocator()` API：XPath 分段 + frameLocator 下降 |
| Playwright MCP | Playwright `frameLocator()` 链式调用 | 原生支持 |

**Cross-origin iframe：** Same-Origin Policy 阻止 JS 访问跨域 iframe 内容。Browser-Use 和 Stagehand 通过为每个 frame origin 创建独立 CDP session 绕过，使用浏览器自身的进程间通信。

**已知限制：** Closed shadow root 内的 iframe 无法用 Playwright 点击（open feature request #23047）。

#### Shadow DOM

- Playwright CSS/text locator 默认穿透 **open** shadow DOM
- XPath **不穿透** shadow root
- **Closed** shadow root 是硬限制——所有系统都无法访问
- Browser-Use 显式例外处理：shadow DOM 中缺少 CDP snapshot 的 form 元素（input, button, select, textarea, a）被强制包含

### 2.7 SPA 交互特有挑战

**Client-side routing：** URL 变化不触发 `page.waitForNavigation()`。Browser-Use 的 `DOMWatchdog` 监控 CDP 事件而非导航事件。

**React/Vue/Angular 异步渲染：** 状态变化后 DOM 可能延迟几毫秒更新。没有框架有框架特定处理——全部依赖通用 DOM 稳定检测。

**Optimistic UI：** React `useOptimistic` 等机制在异步操作完成前显示临时数据。DOM 可能显示暂时不正确的信息。

### 2.8 完整行动词汇表

跨系统统计后的完整行动空间（~25 种）：

| 分类 | 行动 | 优先级 |
|------|------|--------|
| **核心（所有系统）** | click, fill/input_text, scroll_down, scroll_up, select_option, press_key, go_to_url | P0 |
| **重要（多数系统）** | go_back, hover, wait_for, screenshot, get_state/snapshot | P1 |
| **扩展（部分系统）** | go_forward, refresh, check/uncheck, double_click, open_tab, switch_tab, close_tab | P2 |
| **罕见（少数系统）** | drag, clipboard, right_click, long_press, upload_file | P3 |

**不需要的（侦察场景）：** drag, clipboard, double_click, long_press。

## 三、提取：agent 怎么收集数据

### 3.1 提取方法优先级层级（业界共识 + 量化证据）

```
1. 嵌入式 JSON      ← 最高价值、最低成本（~0 cost，95%+ 可靠）
   __NEXT_DATA__, JSON-LD, application/json scripts, 框架状态变量
   通常包含页面的完整数据，比 DOM 中可见的更全

2. 网络捕获的 API    ← 干净的结构化数据（browse 时已被动捕获）
   暴露站点的真实数据模型

3. 直接 API 调用     ← 可重放、可分页（10-100x 快于 DOM 抓取）
   用 httpx/curl 直接调用已发现的端点
   支持翻页、参数化查询

4. DOM 抓取         ← 最常用但最脆弱
   CSS selector / XPath / AX tree
   依赖页面结构，改版即失效

5. RSS/Atom/Sitemap  ← 预结构化，常被忽略
6. 文件下载          ← CSV/JSON 导出
7. GraphQL 内省      ← 如果检测到 GraphQL 端点
8. LLM 驱动提取      ← 最昂贵，fallback
9. 截图 + 视觉模型   ← 非 DOM 内容的最后手段
```

**成本量化（来自 Zyte 分析）：**

| 方法 | 成本 | 速度 | 可靠性 |
|------|------|------|--------|
| 嵌入 JSON | ~0 | 毫秒 | 95%+（数据存在时） |
| API replay | ~0/call | 毫秒 | 90%+（API 稳定时） |
| CSS/XPath | ~0 | 快 | 脆弱——站点改版即失效 |
| LLM 提取 | 5k-10k tokens/page | 秒级 | 高准确但昂贵 |
| 视觉提取 | 10k+ tokens/screenshot | 最慢 | 最后手段 |

**来源：** [zyte.com/blog/agentic-web-scraping/](https://www.zyte.com/blog/agentic-web-scraping/), 多方综合。

### 3.2 嵌入 JSON 提取——完整实现路径

#### Next.js Pages Router

```python
# 检测
script = page.query_selector('#__NEXT_DATA__')
if script:
    data = json.loads(await script.inner_text())
    page_props = data['props']['pageProps']
```

体量：通常 30KB-500KB；超过 1MB 说明站点在发送过多数据。

#### Next.js App Router (RSC Flight Data)

更复杂——数据分布在多个 `self.__next_f.push()` 调用中，行分隔协议，payload 可能跨多个 push 分割。

**推荐工具：** `njsparser` Python 库统一处理两种格式。

检测方法：GET + `RSC: 1` header，测试响应匹配 `/^\d+:["IHL"]/`。

#### JSON-LD

```python
scripts = page.query_selector_all('script[type="application/ld+json"]')
for s in scripts:
    data = json.loads(await s.inner_text())
    # @type: Product, Article, Event, Recipe 等
```

约 40% 的电商站点使用 schema.org。提供干净、标准化的数据，但仅限站点选择暴露的 SEO 信息。**补充源，非主要源。**

Python 库 `extruct` 一次性提取 JSON-LD, Microdata, RDFa, Open Graph, Microformats, Dublin Core。

#### 通用发现算法

Scrapfly 的方法：对页面源码中每个 `{` 字符尝试 `json.JSONDecoder().raw_decode()`。80-85% 可靠。

**来源：** [trickster.dev](https://www.trickster.dev/post/scraping-nextjs-web-sites-in-2025/), njsparser GitHub, [scrapfly.io](https://scrapfly.io/blog/posts/how-to-scrape-hidden-web-data)。

### 3.3 网络捕获提取——"Capture during Browse, Replay Later"

#### 捕获实现

```python
captured_apis = []

async def on_response(response):
    if response.status == 200:
        content_type = response.headers.get("content-type", "")
        if "application/json" in content_type:
            body = await response.body()
            captured_apis.append({
                "url": response.url,
                "method": response.request.method,
                "headers": dict(response.request.headers),  # 含 auth token
                "body_preview": body[:2000].decode("utf-8", errors="replace"),
            })

page.on("response", on_response)  # 必须在 page.goto() 之前注册
```

#### 从捕获到重放

1. 捕获请求的完整 headers（含 Authorization、Cookie、X-* 自定义 header）
2. 逐步去除 header 找到最小必需集
3. 识别分页参数（offset/limit, page, cursor）
4. 构造重放请求

**关键洞察：** "大多数动态站点从内部 API 请求数据——跟踪网络请求找到 JSON 响应，复制 URL，直接抓 API，更快、更干净、更不容易失效。"

#### GraphQL 特殊处理

- 端点通常是 POST 到 `/graphql`，body 含 `query`, `variables`, `operationName`
- **Persisted queries**：很多生产站点只接受已知 query hash，只能重放观察到的请求
- 内省（introspection）"很少公开可用"——别指望
- 替代方案：搜索 JS bundle 中的 query 字符串

**来源：** Playwright 文档, [blog.apify.com/reverse-engineer-apis/](https://blog.apify.com/reverse-engineer-apis/), [scrapfly.io](https://scrapfly.io/blog/posts/web-scraping-graphql-with-python)。

### 3.4 API 发现与调用

**系统化流程（Apify 指南）：**

1. 打开 DevTools Network 标签，过滤 Fetch/XHR
2. 与页面交互（滚动、点击、导航），同时监控
3. 搜索所有响应中的已知数据片段
4. 隔离请求：检查 URL、参数、header、响应结构
5. "Copy as cURL"获取完整请求
6. 在终端测试验证
7. 最小化 header
8. 过滤同域 API 发现所有端点

**robots.txt 的意外价值：** `Disallow` 规则可能暴露 API 端点。Instagram 的 robots.txt 列出了 `/api/`, `/publicapi/`, `/query/`, `/logging/`。

**Sitemap 提供完整 URL 清单**，常比导航爬取更高效。但可能不完整，需与实际站点交叉验证。

**来源：** [blog.apify.com/reverse-engineer-apis/](https://blog.apify.com/reverse-engineer-apis/), [docs.apify.com/academy](https://docs.apify.com/academy/api-scraping/general-api-scraping/locating-and-learning)。

### 3.5 DOM 提取——当前技术水平

#### Stagehand：Schema 驱动

```javascript
const result = await stagehand.page.extract({
    instruction: "Extract product details",
    schema: z.object({
        name: z.string(),
        price: z.number(),
        rating: z.number().optional(),
    })
});
```

用 AX tree + LLM 返回类型化结构数据。

#### AgentQL：语义查询语言

```
{
    products[] {
        name
        price(integer)
        rating(number out of 5 stars)
    }
}
```

AI 语义理解页面元素，按意义定位而非位置。自愈性强。

#### Crawl4AI：多策略

4 种 LLM-free 策略：
- `JsonCssExtractionStrategy`：Schema + CSS selectors
- `JsonXPathExtractionStrategy`：Schema + XPath
- `RegexExtractionStrategy`：预编译模式（email/phone/URL/date/currency）
- `CosineStrategy`：嵌入向量余弦相似度

加 LLM 策略：`LLMExtractionStrategy`——发内容给模型 + 可选 schema。

**自动 schema 生成**：传样本 HTML + query 给 LLM 一次，获得可复用的 CSS/XPath schema。

#### CSS vs XPath vs AX Tree

| 方式 | 优势 | 劣势 | 最佳场景 |
|------|------|------|---------|
| CSS | 快、语法简洁、浏览器优化 | 只能向下遍历，不能按文本匹配 | class/ID/属性选择 |
| XPath | 双向遍历、文本匹配、属性访问 | 浏览器中更慢、更冗长 | 复杂嵌套、基于文本选择 |
| AX Tree | 语义化、抗 UI 变化、过滤噪声 | 需要 CDP，不是所有元素有可访问性信息 | AI 驱动提取、弹性自动化 |

**来源：** Stagehand 文档, AgentQL 文档, Crawl4AI 文档, [scrapingbee.com](https://www.scrapingbee.com/blog/xpath-vs-css-selector/)。

### 3.6 分页与无限滚动

#### 分页类型与检测

| 类型 | 检测信号 | 终止信号 |
|------|---------|---------|
| Next 按钮 | 分页容器中含 "next"/箭头的 `<a>` | href 变为 `#` 或元素消失/disabled |
| 页码 | 分页容器中的数字链接 | 到达最后页码 |
| 无限滚动 | URL 不变，滚动时有 XHR 请求 | `scrollHeight` 不变，API 返回空 |
| Load More 按钮 | 触发 XHR 的按钮 | 按钮消失，response 中 `remaining: 0` |
| Cursor 分页 | API URL 含 `cursor`/`after` 参数 | `next_cursor: null` 或 `has_next: false` |

#### 分页图模型（ScrapingAnt）

把分页建模为有向图：节点 = 分页状态（URL + params + cursor），边 = 导航转换。防止无限循环：

```
停止条件：
- 节点 key 已访问（结构循环）
- 内容哈希与最近 N 页重复（内容循环）
- 最近 M 页新项目 < 阈值（收益递减）
```

#### 虚拟化列表（最难情况）

`react-window` / `react-virtualized` 只在 DOM 中渲染可见项。不可见项**不存在于 DOM**。策略：
1. 增量滚动 + 逐段提取 + 去重
2. **拦截底层 API 调用**（最佳——绕过虚拟化）
3. 注入 JS 访问 React fiber tree 获取完整数据

**关键洞察：分页检测最好通过网络捕获。** 观察滚动/点击时的 XHR 请求暴露底层分页 API，然后直接重放 API 修改参数——比浏览器模拟高效几个数量级。

**来源：** [scrapingant.com/blog/pagination-as-a-graph](https://scrapingant.com/blog/pagination-as-a-graph-modeling-infinite-scroll-and-loops), [oxylabs.io/blog/pagination](https://oxylabs.io/blog/pagination-in-web-scraping)。

### 3.7 提取质量验证

#### 坏提取的检测信号

| 信号 | 含义 |
|------|------|
| 字段级 null rate 突增（<1% → >50%） | Selector 失效（schema drift） |
| 记录数异常（期望 100 条，得到 3 条） | 部分提取 |
| HTML 伪影（`<br>`, `&amp;`, `&nbsp;`） | DOM 清理不完整 |
| 类型不匹配（boolean 字段含字符串 "False"） | 解析错误 |

#### 验证方法

- **Schema 验证**：JSON Schema / Pydantic 强制类型、格式（regex）、必填字段
- **统计分析**：数值字段标准差检测异常值；值频率分析（`value_counts()`）
- **备选 selector**：关键字段定义二级/三级提取路径——主 CSS selector 返回 null 时尝试备选

#### 去重策略

- URL 级：去除 tracking 参数（`utm_*`, session ID），排序 query params，标准化协议
- 内容级精确：SHA-256 of normalized text
- 内容级近似：SimHash, MinHash, n-gram shingles + Jaccard（阈值 ≥ 0.95）
- 实体级：确定性 key（SKU, ISBN）+ 模糊匹配（名称相似度）

**来源：** [scrapingant.com/blog/building-a-web-data-quality-layer](https://scrapingant.com/blog/building-a-web-data-quality-layer-deduping-canonicalization), [zyte.com/blog/guide-to-web-data-extraction-qa](https://www.zyte.com/blog/guide-to-web-data-extraction-qa-validation-techniques/)。

### 3.8 方法自动选择

没有生产系统完整实现自动方法选择。但从多个来源浮现的模式是**效率优先的逐级升级**：

```
1. 检查嵌入 JSON       → 有？直接解析（无需渲染，无 LLM 成本）
2. 检查网络 API 捕获    → 有 JSON API 响应？直接重放
3. 主动 API 发现        → robots.txt, sitemap, 常见路径探测, 内省
4. DOM 提取 + selectors → CSS/XPath/schema-based
5. LLM 驱动 DOM 提取    → 发页面内容给 LLM + schema
6. 视觉/截图提取        → 最后手段
```

Zyte 的架构确认这个顺序："inspect network activities, search for direct API opportunities, craft reliable selectors, spot embedded JSON...invoke browser rendering only when necessary."

**混合优化：** Crawl4AI 的 `generate_schema()` 让 LLM 一次性生成可复用的 CSS/XPath schema，结合 AI 灵活性和传统速度。

## 四、计算与 CodeAct：agent 怎么处理数据

### 4.1 CodeAct 范式——原始证据

**论文：** "Executable Code Actions Elicit Better LLM Agents"（Xingyao Wang et al., ICML 2024）

**核心机制：** agent 生成 Python 代码而非 JSON tool call。代码由 Python 解释器执行，stdout/stderr 作为 observation 返回。

**关键数据点：**
- GPT-4 on M3ToolEval：**CodeAct 74.4% vs Text 53.7%**（+20.7pp），减少 2.1 次交互
- 17 个模型测试：CodeAct 对强模型效果显著，弱模型差距小（开源最佳 Lemur-70b 仅 13.4%）
- 自我调试是关键机制：Python traceback 提供自然的调试信息

**改进的根本原因：** 代码天然支持控制流。一个 `for` 循环在一次 action 中对所有输入应用相同操作；JSON/text 每个输入需要单独一次 action。

**来源：** [arXiv:2402.01030](https://arxiv.org/abs/2402.01030), [xwang.dev/blog/2024/codeact/](https://xwang.dev/blog/2024/codeact/)。

### 4.2 生产系统都是混合的

**没有成功的系统用纯 CodeAct 或纯 Tool-calling。**

| 系统 | 结构化工具 | 代码执行 | 混合方式 |
|------|-----------|---------|---------|
| **Manus** | 20+ tool calls（browser_navigate, browser_click 等） | `shell_exec` 写文件+执行 | 工具 API 中介所有 action，代码通过 shell_exec 补充 |
| **OpenHands** | browser, str_replace_editor | `execute_bash` + `execute_ipython_cell`（持久 Jupyter） | CodeActAgent 委托 BrowsingAgent 处理 web |
| **Claude Code** | read, write, edit, glob, grep | bash（持久 shell session） | 结构化工具处理常见文件操作，bash 处理其他一切 |
| **SWE-Agent** | 自定义 ACI 命令（file viewer/editor/search） | Docker shell | 自定义命令 > 原始 shell（+10.7pp ablation） |

**Anthropic PTC（Programmatic Tool Calling）：** 官方 CodeAct 实现。Claude 在沙箱容器中写 Python 代码，代码中调用 tool 函数时暂停，外部提供结果后恢复。中间结果留在容器——只有 `print()` 输出进入 context。

```python
# Agent 写的代码
results = []
for url in urls:
    page = await browse(url)  # tool 调用在这里暂停
    results.append(page.title)
print(f"Found {len(results)} pages")  # 只有这进入 context
```

**PTC 性能：** BrowseComp/DeepSearchQA benchmark 上是"完全解锁 agent 性能的关键因素"。Token 从 43,588 降至 27,297（-37%），知识检索从 25.6% 升至 28.5%。

**`allowed_callers` 机制：** 工具可以是 `["direct"]`（传统）、`["code_execution_20260120"]`（仅 PTC）、或两者都有。

**来源：** Manus leaked system prompt, OpenHands CodeAct README, Claude Code 文档, SWE-agent 论文, [Anthropic PTC 文档](https://platform.claude.com/docs/en/agents-and-tools/tool-use/programmatic-tool-calling)。

### 4.3 CodeAct vs Tool-calling 对比

| 维度 | Tool Calling | CodeAct |
|------|-------------|---------|
| Action 粒度 | 一次 LLM turn 一个 tool | 一次 turn 多个操作 |
| LLM 推理轮次 | 每个 tool call 一次 | 每个 code block 一次（含多个调用） |
| Context 效率 | 所有中间结果在 context | 结果留在 runtime，只有 print 进入 context |
| 控制流 | 无循环/条件（LLM 每步推理） | Python 原生 for/if/try-except |
| 训练数据对齐 | JSON schema 是合成格式，训练数据稀疏 | Python 代码在训练数据中极丰富 |
| 可观测性 | 每个 action 是离散、可日志的事件 | Code block 是不透明单元 |
| 错误处理 | 框架按 action 捕获错误 | Agent 必须在代码中处理（try/except） |
| 安全面 | 最小（schema 验证输入） | 广（任意代码执行） |

**CodeAct 明显更好的场景：**
- 多步数据处理（过滤、聚合、转换）
- 迭代操作（检查 N 个项目，处理列表）
- 计算（数学、字符串操作、数据结构）
- 工具组合（多个工具输出合成一个结果）

**Tool-calling 明显更好的场景：**
- 浏览器交互（需要框架管理 SPA 检测、等待、JS 执行等复杂状态）
- 需要逐 action 日志和审计的场景
- 弱模型（开源模型 CodeAct 差距小）

### 4.4 Web Agent 的最佳混合模式

**"Beyond Browsing" 论文（WebArena）量化证据：**

| Agent 类型 | 准确率 |
|-----------|--------|
| 仅浏览 | 14.8% |
| 仅 API/代码 | 29.2% |
| **混合** | **38.9%** |

混合 agent 在 77.7% 的任务中同时使用了两种模态。

**推荐模式：高层工具 + 代码做计算**

- `browse(url)` 作为 tool call：处理导航、SPA 检测、自动等待、页面摘要——这些复杂度在生成代码中很难搞对
- `interact(action, target)` 作为 tool call：处理 click/fill/scroll + 自动重试和元素解析
- `extract(script)` 作为 tool call：在浏览器上下文中执行 JS，保存样本，返回摘要
- `bash` 做计算：数据处理、API 调用、文件操作、条件逻辑

浏览器交互保持可靠（框架管理），数据处理给 agent 完全计算自由。

### 4.5 Vercel d0 实验——深度分析

**背景：** 内部 text-to-SQL agent，16 个专用工具花了几个月构建。删掉 80% 后只留 `ExecuteCommand`（bash）和 `ExecuteSQL`。

| 指标 | 之前（16 工具） | 之后（2 工具） |
|------|---------------|--------------|
| 成功率 | 80%（4/5） | 100%（5/5） |
| 执行时间 | 274.8s | 77.4s（3.5x 快） |
| Token | ~102k | ~61k（-37%） |
| 步数 | ~12 | ~7（-42%） |

**关键细节：** sandbox 部署了 YAML/Markdown/JSON 语义层文件。Claude 用 `grep`/`cat`/`find`/`ls` 发现 schema——和专用工具做的一样，但通过通用命令。

**领域差异：** text-to-SQL 是结构化、有界的领域。Web 侦察是非结构化、无界的。d0 的教训部分适用：不要用过多专用工具过度约束 agent，但浏览器管理的复杂度确实需要框架级处理。

**核心洞察：** Vercel 发现他们在"替 model 思考"——通过预过滤上下文和验证逻辑。现代 LLM 善于在原始信息上推理。给 agent 文件系统访问而非精心策划的工具，让模型用训练数据中已有的技能。

**来源：** [vercel.com/blog/we-removed-80-percent-of-our-agents-tools](https://vercel.com/blog/we-removed-80-percent-of-our-agents-tools)。

### 4.6 持久状态的重要性

CaveAgent（双流架构）：语义流 + 运行时流。变量跨 turn 持久化。结果：比无状态方案高 10.5pp 成功率，低 28.4% token。

OpenHands 的 `execute_ipython_cell` 基于 Jupyter kernel，变量跨调用存活。

**对我们的含义：** 如果采用代码执行，持久 Python session（如 IPython kernel）在容器内运行，变量跨 turn 存活，会显著提升多步任务效能。我们已经在 Docker 中运行。

### 4.7 bash 和 execute_code 应该合并

**全部证据指向同一方向：**
- `execute_code("print('hello')", "python")` ≡ `bash("python3 -c \"print('hello')\"")`
- 多行脚本：bash 可以写临时文件或用 heredoc
- 生产系统：Manus 只有 `execute_command`，Claude Code 只有 Bash，SWE-Agent 只有 bash
- 减少工具 = 减少决策负担 = 更好表现

**来源：** d0 实验, mini-SWE-Agent（~100 行 Python，只有 bash，SWE-bench 74%+）。

## 五、鲁棒性：agent 怎么应对意外

### 5.1 Cookie Consent 自动处理

**分层策略（从可靠到兜底）：**

| 层级 | 方案 | 覆盖度 | 可靠性 |
|------|------|--------|--------|
| 1 | `@duckduckgo/autoconsent` | 100+ CMP（OneTrust, Cookiebot, Quantcast, TrustArc, Google） | 高 |
| 2 | 启发式 DOM 检测 | 高 z-index + position:fixed + cookie/consent 文本 | 中 |
| 3 | CSS 注入隐藏 | `[class*="cookie"]`, `[id*="consent"]` 等 → `display:none` | 低（不解除滚动锁定） |
| 4 | 已知站点预设 Cookie | 已知 CMP 直接注入 consent cookie | 最高（但需要 per-site 知识） |

**关键细节：**
- Shadow DOM banner：Playwright CSS/text locator 默认穿透 open shadow DOM
- 滚动锁定 banner：CSS 注入不够，需要移除 `overflow: hidden` + 删除 overlay 元素
- Browser-Use 有已知的 popup 处理问题（issue #1996）：cookie prompt、newsletter modal 等导致 agent 卡住

**推荐：** 作为**基础设施层**自动处理，agent 不应该感知到 cookie banner。`autoconsent` 作为第一层，启发式作为兜底。

**来源：** [github.com/duckduckgo/autoconsent](https://github.com/duckduckgo/autoconsent), [blog.apify.com](https://blog.apify.com/how-to-block-cookie-modals/), Browser-Use issue #1996。

### 5.2 Overlay / Modal 检测与处理

**JS dialog vs DOM modal 的关键区别：**

| 类型 | 检测方式 | 处理方式 | 阻塞性 |
|------|---------|---------|--------|
| JS dialog（alert/confirm/prompt） | `page.on('dialog')` | `dialog.accept()` / `dialog.dismiss()` | 冻住整个页面 |
| DOM modal | 正常 DOM 交互 | 点击关闭/按 Escape/移除元素 | 可能锁定滚动 |

**遮挡检测：** 在每次交互前用 `document.elementFromPoint(x, y)` 检查目标元素是否被遮挡。如果返回的不是目标元素或其后代，说明有遮挡。

Browser-Use 的实现：检查双向包含关系 + label-input 关联（向上 3 层祖先）。被遮挡时回退 JS click。

**Playwright 的 `addLocatorHandler()`：** 注册一个函数，当指定的遮挡 overlay 出现时自动执行（dismiss 后再继续 action）。

**推荐：**
- Dialog → 基础设施层 `page.on('dialog')` 自动处理
- DOM overlay → 交互前遮挡检测 + 自动尝试关闭

### 5.3 Infinite Scroll 与 Lazy Loading

**虚拟化列表是最难的情况：** DOM 中只有可见项。只能通过 API 拦截或 React fiber 注入获取完整数据。

**停止标准（防止无限循环）：**
- 连续 N 次滚动无新内容（通常 3-5 次）
- 最大 unique items 数
- 最大深度/步数
- 边际收益低于阈值

**最佳策略始终是 API 拦截**而非滚动模拟。

### 5.4 认证与登录墙

**检测信号：**
- HTTP 401/403
- 重定向到含 "login", "signin", "auth", "sso" 的 URL
- 页面含 `<input type="password">`
- "Sign in to continue" 文本
- 内容明显少于预期（软 paywall）

**"Call Human" 模式已成熟：**

| 系统 | 实现方式 |
|------|---------|
| OpenAI Operator | Takeover Mode：暂停 agent，人类接管浏览器，完成后交还。接管期间不截图、AI 不可见 |
| 1Password | Secure Agentic Autofill：检测登录页 → 1Password 识别凭据 → 生物识别授权 → 加密注入。凭据**不进入 LLM context** |
| Anthropic CU | 敏感认证步骤提示人类 |

**Session 持久化：** Playwright `context.storageState()` 保存 cookie + localStorage 到 JSON。`browser.newContext(storageState='state.json')` 恢复。

**推荐：** MVP 阶段记录登录墙为 observation（"此位置需要认证"），尝试替代路径（API、缓存版本）。不尝试绕过。

**来源：** [openai.com/index/introducing-operator/](https://openai.com/index/introducing-operator/), [developer.1password.com](https://developer.1password.com/docs/agentic-autofill/)。

### 5.5 反爬对抗

**Camoufox 覆盖的：** C++ 级别指纹伪装（navigator, WebGL, Canvas, AudioContext, screen, WebRTC）。

**Camoufox 覆盖不了的：**

| 检测方式 | 说明 | Camoufox 能否绕过 |
|---------|------|-------------------|
| TLS 指纹（JA3/JA4） | TLS 握手 cipher suites + extensions 产生独特指纹 | 真实浏览器引擎，TLS 指纹正确 ✓ |
| 行为分析（Akamai/DataDome） | 鼠标轨迹、滚动模式、键盘节奏实时分析 | 含 Bezier 曲线模拟 ✓ |
| IP 声誉 | 数据中心 IP 直接标记 | **无法解决** ✗ |
| SpiderMonkey 引擎检测 | WAF 测试 Firefox 引擎特有行为 | **无法伪装** ✗ |
| 指纹一致性 | Windows UA + Apple GPU = 标记 | **2026 年有退化** ✗ |

**Camoufox 2026 年状态：** 有维护间隙，一些新发现的指纹不一致性已降低效果。`puppeteer-extra-stealth` 已于 2026 年 2 月弃用。

**Rate Limiting 处理：**
- 检测：HTTP 429, `Retry-After` header, `X-RateLimit-*` headers, 响应变慢
- 处理：指数退避（1s, 2s, 4s, 8s...）+ jitter

**来源：** [github.com/daijro/camoufox](https://github.com/daijro/camoufox), [datadome.co](https://datadome.co/anti-detect-tools/camoufox/), [scrapfly.io](https://scrapfly.io/blog/posts/how-to-bypass-akamai-anti-scraping)。

### 5.6 导航边界情况

| 情况 | 处理方式 |
|------|---------|
| HTTP 404 | 记录 URL 无效，不重试 |
| HTTP 403 | 可能是访问限制或反爬，记录并尝试替代 |
| HTTP 429 | Rate limited，退避重试 |
| HTTP 5xx | 服务器错误，可能是暂时的，退避重试 |
| 超时 | 用 `waitUntil: 'domcontentloaded'` 而非 `'load'`；增加慢站超时 |
| 下载链接 | `page.waitForEvent('download')` + `download.saveAs(path)` |
| `javascript:void(0)` / `href="#"` | 依赖 onclick handler，不触发导航；不要用 `waitForNavigation()` |
| Referrer 要求 | 通过站内导航自然产生正确 referrer；避免直跳深链接 |

### 5.7 内存与稳定性

**浏览器长时间运行约 2-3% 崩溃率。** 缓存页面 20 分钟可消耗 400MB+。页面 reload **不释放内存**——只有关闭再创建才释放。

**最佳实践：**
1. 不再需要的页面立即 `page.close()`
2. 定期创建新 browser context 而非复用整个 session 的一个
3. 监控内存使用，接近限制时主动回收
4. 清理 event listener 和 timer

**崩溃恢复：** World Model 在 PostgreSQL 中天然作为 checkpoint。浏览器崩溃 mid-session，ReconPlanner 可以启动新 session，所有发现通过 World Model 保留。

**来源：** Playwright issue #15400, [markaicode.com](https://markaicode.com/playwright-mcp-browser-crash-recovery-patterns-2025/)。

### 5.8 多标签管理

- 每个 tab 是 BrowserContext 中的一个 `Page` 对象
- `context.on('page', handler)` 捕获所有新标签
- `page.waitForEvent('popup')` 捕获点击触发的弹出窗口
- **始终关闭不用的 tab**——每个 open page 消耗内存
- OAuth 流程通常打开弹窗 → 交互 → 弹窗关闭 → 切回原页面

## 六、跨领域综合结论

### 6.1 关键设计原则（有证据支撑）

| 原则 | 证据 | 来源 |
|------|------|------|
| **感知质量 > 推理能力** | 仅优化观察空间效果比改进架构高 161% | AgentOccam (ICLR 2025) |
| **DOM 层级是最重要特征** | 丢弃层级一致降低性能；Markdown+HTML 混合最佳 | D2Snap (arXiv:2508.04412) |
| **混合 > 纯模态** | 混合 38.9% vs 浏览 14.8% vs API 29.2% | Beyond Browsing (ACL 2025) |
| **所有成功系统都是混合的** | Manus, OpenHands, Claude Code, SWE-Agent 无一纯 CodeAct 或纯 tool-call | 多方验证 |
| **每个工具必须证明存在价值** | 去掉 80% 工具后 3.5x 快 | Vercel d0 |
| **变化观察是关键创新** | 交互后告诉 agent 发生了什么，省掉 interact→browse 双步 | Agent-E |
| **不跨页面维持元素引用** | 所有系统每次 action 后重建快照 | 业界共识 |
| **嵌入 JSON > API > DOM** | 成本递增，脆弱性递增 | Zyte, Scrapfly, 多方验证 |
| **API 拦截 >> 滚动模拟** | 虚拟化列表只能走 API | 多方验证 |
| **基础设施消除沉默故障** | dialog/popup/SPA 等待不应到达 agent | Cookie banner, dialog, 多方验证 |

### 6.2 必须是专用工具的能力

| 能力 | 原因 |
|------|------|
| 浏览器页面观察 | 需要 Playwright/CDP API，非文本交互。含 AX tree、网络捕获、嵌入数据检测 |
| 浏览器元素交互 | 需要 Playwright/CDP API + 元素解析 + 遮挡检测 + 回退链 |
| 浏览器内 JS 提取 | 需要 `page.evaluate()`，继承 session cookies |
| World Model 写入 | 需要 DB + 内存模型同步 |
| World Model 读取 | 需要格式化呈现 |
| 推理空间 | tool-calling 模式下的行动前推理（Anthropic 证实价值） |

### 6.3 应该交给 bash 的能力

| 能力 | 原因 |
|------|------|
| 直接 HTTP 请求 | `curl` / `httpx` 足够，无需专用工具 |
| 数据分析处理 | Python/jq 管道，灵活性无法用专用工具替代 |
| 文件操作 | 读写中间文件，bash 自然支持 |
| 代码执行 | `python3 -c` 或写临时脚本 |
| sitemap/robots.txt | `curl` 直接获取 |
| API 参数探索 | 循环 + curl |
| 上网搜索资料 | `curl` + search API |

### 6.4 应该内建于基础设施的能力

| 能力 | 方案 |
|------|------|
| Dialog 自动处理 | `page.on('dialog', handler)` — auto-accept alert/confirm, auto-dismiss prompt |
| Cookie banner 自动处理 | `@duckduckgo/autoconsent` + 启发式兜底 |
| 新标签检测 | `context.on('page')` 自动追踪 |
| SPA 智能等待 | MutationObserver settle（1.5s 静默）+ 硬超时 |
| CSP 绕过 | `bypass_csp=True` |
| 网络请求被动捕获 | `page.on('response')` 在导航前注册，数据通过 browse 结果呈现 |
| 导航错误处理 | 捕获 `page.goto()` 异常，返回结构化错误 |
| 遮挡预检 | 交互前 `elementFromPoint()` 检查 |
| 浏览器崩溃恢复 | 检测 CDP 断开 + 重连 |
| 内存管理 | 定期回收 context，关闭不用的 page |
