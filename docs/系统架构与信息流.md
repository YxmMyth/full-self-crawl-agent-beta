# 系统架构与信息流

> 状态：共识稿 v2
> 日期：2026-03-24
> 前置：CLAUDE.md
> 调研基础：OpenHands, ESAA, Manus, Devin, Skyvern, AgentFold, JetBrains Complexity Trap, Voyager, MemGPT/Letta, Chroma context rot study

---

## 一、系统的根本目标

给定域名 + 自然语言需求，**自主探索**网站结构，**建构理解**数据分布，**采集样本**验证理解。

最终形态是对目标网站进行**完整的 L1-L4 建模**：

- **L1 站点结构**：站点有哪些 URL pattern、它们如何连接
- **L2 数据分布**：每个 pattern 有什么数据、什么格式、多少量
- **L3 需求映射**：需求对应哪些 pattern、路径是否通畅
- **L4 样本**：实际提取的数据样本

这不是爬虫。本系统的核心能力是**在未知环境中建立认知**。产出不是"数据"而是"对数据的理解 + 证据样本"。

## 二、两层架构（共识）

```
ReconPlanner（战略层）
  拥有 World Model 的全局视图
  职责：评估进展、决定方向、编译任务书
  不决定 agent 去哪个 URL、不微管理探索顺序
      │
      │  briefing（编译后的任务书）
      ▼
Agent Session（执行层）
  独立 agent：自己的 context window、工具集、步数预算
  职责：导航、观察、交互、提取、记录发现
  录制 Agent 并行维护 observations（CRUD），transcript 是不可变记录
  通过 read_world_model 按需查询已有知识
```

### 为什么是两层？

单层 agent 的 context 在 ~30 步后饱和（JetBrains：环境输出占每轮 token 的 84%）。一个 agent 无法同时保持战略视野和操作细节。两层的本质是**认知分工**：Planner 做全局评估，Session 做局部执行。

### Session 即 Agent

每个 Session 是完全独立的 agent 实例：
- **独立 context**：不继承前序 session 的 tool call 历史
- **独立预算**：固定步数上限
- **共享知识**：通过 briefing 获取前序知识，通过 read_world_model 按需查询
- **独立产出**：observations 写入 World Model

Session 之间不直接通信。所有跨 session 的知识传递都经过 World Model。

## 三、信息流（共识）

### Session → World Model（实时写入）

录制 Agent 与执行 Agent 实时并行，维护 observations（可创建/更新/合并/删除）。Transcript（JSONL）是不可变的原始记录。并发写入安全性由 DB 事务保证。

### World Model → Planner（格式化后的 observations）

Planner 的 LLM 调用直接读格式化后的 observations——**不经过预先的 LLM 摘要/投影**。程序化步骤仅负责从 DB 取数据并格式化成文本（数据搬运），所有"理解、评估、决策"交给 LLM 在同一次调用中完成。

格式化采用**分层衰减**（调研共识模式）：
- 当前 session 的 observations：完整展示
- 前序 session 的 observations：简略展示（类型 + 关键信息）
- 越远越简略，总量控制在 LLM 有效处理范围内

### World Model → Session（任务书 + 按需查询）

两个渠道：
- **Briefing**（推送）：Planner LLM 编译的任务书，包含方向 + 相关 location 的原始 observations
- **read_world_model**（拉取）：Agent 主动查询，默认返回索引视图，可 drill-down 到具体 location

## 四、Planner 循环（共识）

```
Session 结束
  │
  ├─ 程序化：从 DB 取 observations + sessions，格式化成文本
  │    （唯一的程序化步骤——数据搬运，不做理解/判断）
  │
  ├─ LLM 调用 1：evaluate_and_decide
  │    输入：格式化后的 observations（分层衰减）+ session 历史 + requirement
  │    输出：DONE（原因）
  │          或 CONTINUE + 方向 + relevant locations
  │    （综合理解和决策在同一次调用中完成，无序列化边界的信息损失）
  │
  ├─ [feature-gated] spawn Verification Subagent（仅在 DONE 时）
  │    输入：WM 快照 + requirement + DONE 理由
  │    独立 tool-use 循环：read_world_model + bash + think
  │    输出：VERDICT: PASS / FAIL / PARTIAL + gaps
  │    if PASS: 真正结束
  │    if FAIL/PARTIAL: 覆盖 DONE → CONTINUE，gaps 进入 briefing
  │    （可通过 VERIFICATION_SUBAGENT_ENABLED=false 完全禁用，禁用后 DONE 直接生效）
  │
  └─ LLM 调用 2：generate_briefing（仅在 CONTINUE 时）
       输入：方向 + relevant locations 的原始 observations + [可选] verification gaps
       输出：具体任务书（含已知信息、探索方向、具体起点）
```

### 为什么是两次 LLM 调用而不是一次？

evaluate_and_decide 和 generate_briefing 需要**不同粒度的信息**：

- evaluate_and_decide 需要**全局广度**：所有 locations 的概况，做"够了吗？往哪走？"的判断
- generate_briefing 需要**局部深度**：特定 locations 的原始 observations（URL、selector、字段名、失败原因）

把两者合并意味着要在一次调用中塞入"全部 location 概况 + 特定 location 原始细节"，context 负担更大且 relevant locations 在 evaluate_and_decide 之前未知。

### 为什么没有独立的 reflect 步骤？

之前的设计有三步：reflect → decide → briefing。但 reflect 和 decide 需要相同的输入（observations + session history），拆成两步会在序列化边界损失信息——reflect LLM 形成的理解被压缩成文本，decide LLM 只能读到这段文本而非原始数据。合并后，LLM 在同一个 context 中完成综合理解和决策。

### 为什么没有程序化的 outcome_summary / location_states？

之前设计了程序化的 outcome_summary（统计 session 产出）和 location_states（计算 frontier 状态）。但：

1. 程序化处理容易引入 schema mismatch 类 bug（如 `obs.raw.get("type")` 事件导致 9/9 runs 失败）
2. LLM 直接读 observations 能理解内容含义，不依赖字段名约定
3. "智能优先"硬约束：理解和判断类工作交给 LLM，程序只做数据搬运

程序化步骤的职责收窄到**数据搬运**：从 DB 取数据 → 格式化 → 组装 prompt。

## 五、数据膨胀处理（共识）

observations 随 session 累积不断增长。核心调研发现：

- "Just use bigger context" 不 work——Chroma 测试 18 个前沿模型，全部随 context 增长退化
- ~16K tokens 以内 LLM 决策质量稳定，超过 32K 明显下降
- 分层衰减是 Manus、OpenHands、Claude Code、Cursor 趋同演化的共同模式

### 分层格式化策略

```
当前 session 的 observations → 完整展示
前 1-2 个 session 的 observations → 类型 + 关键信息摘要
更早的 sessions → 每 session 一行总结
Locations 列表 → 始终完整展示（pattern + obs 数量）
```

这是纯程序化的格式化（数据搬运），不涉及 LLM 调用。通过控制每层的展示详细度来控制总 token 量。

### 当规模超出单次调用上限时

如果 locations 膨胀到数十个、observations 累积到数百条，即使分层格式化也可能超过有效 context 范围。此时引入 LLM 预压缩（类似之前的 ES），作为**性能优化而非架构必需**。当前 MVP 规模（10-20 locations，100-200 observations）不需要。

---

## 六、停止条件

1. **evaluate_and_decide 返回 DONE** — LLM 判断侦察充分
2. **Session 预算耗尽** — 达到 max_sessions 上限
3. **知识停滞** — 连续 N 轮 session 没有产出新的有价值 observations

## 七、并发架构（预留）

### Phase 1：串行（MVP）

```
Planner → Session 1 → evaluate_and_decide → Session 2 → ...
```

### Phase 2：并行探索（未来）

```
Planner → evaluate_and_decide 返回 N 个方向
  ├─ Session A (方向 1) ─┐
  ├─ Session B (方向 2) ─┤ 并发执行
  └─ Session C (方向 3) ─┘
  └─ Barrier sync → evaluate_and_decide → 下一轮
```

并发安全由 DB 事务保证（录制 Agent 是各 session 的 Observations 唯一写入者）。从串行到并行的切换点仅在 Planner 主循环。

## 八、设计决策记录

### D1: 理解和判断全部 LLM 化，程序只做数据搬运

所有需要"理解"的工作（综合 observations、评估进展、决定方向、编写任务书）交给 LLM。程序化步骤仅负责从 DB 取数据并格式化成文本。

理由：程序化处理 observations 容易引入 schema mismatch 类 bug；LLM 直接读原始数据更可靠；符合"智能优先"硬约束。

### D2: evaluate_and_decide 合并反思与决策

不设独立的 reflect 步骤。综合理解和战略决策在同一次 LLM 调用中完成，避免序列化边界的信息损失。

理由：每次 LLM 调用完全独立无状态。拆成两步意味着第一步的"理解"必须序列化为文本传给第二步，文本永远是有损压缩。

### D3: Briefing 包含原始 observations

generate_briefing 的输入包含 relevant locations 的原始 observations。Agent 需要精确细节（URL、selector、字段名）来执行任务，这些在任何摘要中都会丢失。

### D4: Location 粒度由 agent 决定

Location 代表"站点上一个位置"，粒度不硬编码。agent 自己决定用 URL pattern（`/tag/{tag}`）还是更具体的形式。系统不强制归纳规则。符合"不预设网站类型"硬约束。

### D5: 数据膨胀用分层格式化应对，不引入预压缩

MVP 规模下，程序化的分层格式化（近的详细、远的简略）足以控制 Planner 输入量。LLM 预压缩（ES）在未来规模超出时作为优化引入，不是架构必需。

### D6: 并发预留，MVP 串行

录制 Agent 是各 session 的 Observations 唯一写入者，DB 事务保证并发安全。从串行到并行只需改 Planner 主循环。
