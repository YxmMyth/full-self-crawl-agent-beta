## 一、系统的根本目标

给定域名 + 自然语言需求，**自主探索**网站结构，**建构理解**数据分布，**采集样本**验证理解。

最终形态是对目标网站进行**完整的 L1-L4 建模**：

- **L1 站点结构**：站点有哪些 URL pattern、它们如何连接
- **L2 数据分布**：每个 pattern 有什么数据、什么格式、多少量
- **L3 需求映射**：需求对应哪些 pattern、路径是否通畅
- **L4 样本**：实际提取的数据样本

这不是爬虫。本系统的核心能力是**在未知环境中建立认知**。产出不是"数据"而是"对数据的理解 + 证据样本"。

## 二、两层架构（共识）

```
ReconPlanner（战略层）
  拥有 World Model 的全局视图
  职责：评估进展、决定方向、编译任务书
  不决定 agent 去哪个 URL、不微管理探索顺序
      │
      │  briefing（编译后的任务书）
      ▼
Agent Session（执行层）
  独立 agent：自己的 context window、工具集、步数预算
  职责：导航、观察、交互、提取、记录发现
  录制 Agent 并行维护 observations（CRUD），transcript 是不可变记录
  通过 read_world_model 按需查询已有知识
```

### 为什么是两层？

单层 agent 的 context 在 ~30 步后饱和（JetBrains：环境输出占每轮 token 的 84%）。一个 agent 无法同时保持战略视野和操作细节。两层的本质是**认知分工**：Planner 做全局评估，Session 做局部执行。

### Session 即 Agent

每个 Session 是完全独立的 agent 实例：
- **独立 context**：不继承前序 session 的 tool call 历史
- **独立预算**：固定步数上限
- **共享知识**：通过 briefing 获取前序知识，通过 read_world_model 按需查询
- **独立产出**：observations 写入 World Model

Session 之间不直接通信。所有跨 session 的知识传递都经过 World Model。

## 三、信息流（共识）

### Session → World Model（实时写入）

录制 Agent 与执行 Agent 实时并行，维护 observations（可创建/更新/合并/删除）。Transcript（JSONL）是不可变的原始记录。并发写入安全性由 DB 事务保证。

### World Model → Planner（Push + Pull）

Planner 是 tool-use agent，通过两种方式获取信息：
- **Push**：spawn_execution / spawn_research 的 tool_result 自动返回摘要（session summary + model_diff + key_findings）
- **Pull**：Planner 调 read_model() 按需查看完整 Semantic + Procedural Model

Planner 不直接读 observations——observations 由 maintain_model LLM 函数提炼进 Model，Planner 只读 Model。

### World Model → Session（任务书 + 按需查询）

两个渠道：
- **Briefing**（推送）：Planner LLM 编译的任务书，包含方向 + 相关 location 的原始 observations
- **read_world_model**（拉取）：Agent 主动查询，默认返回索引视图，可 drill-down 到具体 location

## 四、Planner 循环（共识）

```
Planner tool-use 循环
  │
  ├─ Planner LLM 决定下一个 tool call
  │    工具集（5 个）：spawn_execution / spawn_research / read_model / think / mark_done
  │
  ├─ spawn_execution(briefing) 的内部 pipeline：
  │    1. 启动 Session（执行 Agent + 单例 Recording Agent 协作）
  │    2. Session 停止
  │    3. maintain_model（Python 代码自动触发的 LLM 函数）
  │       输入：current Model + new observations
  │       输出：new Model（写回 DB）+ session summary + model_diff
  │    4. 返回 { summary, model_diff } 给 Planner（Push）
  │
  ├─ read_model() → 返回完整 Semantic + Procedural Model（Pull）
  │
  └─ mark_done(reason) 的内部 pipeline：
       [feature-gated] 自动触发 Verification Subagent
       PASS → { status: "DONE" }
       FAIL/PARTIAL → { status: "blocked", gaps }（Planner 继续循环）
```

### 为什么 Planner 是 tool-use agent 而不是固定 LLM 调用？

之前设计是 evaluate_and_decide + generate_briefing 两次固定调用。改为 tool-use agent 是因为：
- 硬约束 #7（不硬编码控制流）要求决策由 LLM 做
- Research 触发时机不清 → LLM 自主判断比硬编码 signal 机制灵活
- 执行 Agent 和 Research Agent 平级派发，逻辑一致
- 两层都是 tool-use agent，架构对称

### 为什么 maintain_model 不是 Planner 的工具？

之前方案让 Planner 有 update_model 工具。但"宏观决策"和"抽象提炼"是不同抽象层级的工作，塞进同一个 LLM 循环 → LLM 偷懒/乱序。行为教学 + 安全网是补丁，不治本。

maintain_model 是数据通道（observations → model 的提炼），不是控制流（"下一步做什么"的决策）。由 Python 代码在 spawn_execution 返回前自动触发，不违反硬约束 #7。

### 为什么没有程序化的 outcome_summary / location_states？

理由不变：程序化处理容易引入 schema mismatch 类 bug。LLM 直接读 observations 能理解内容含义。"智能优先"硬约束：理解和判断类工作交给 LLM。

## 五、数据膨胀处理（共识）

Model 层本身就是 LLM 的压缩产物——每次 maintain_model 全量重写 Semantic + Procedural Model，保持在字符上限内（Semantic ~8K，Procedural ~6K）。旧 observations 的信息被融入 Model 后，Planner 不再直接读取。

因此**不需要分层衰减格式化**：
- Planner 只读 Model（已压缩）+ 最新 session 的 summary（短文本）
- Model 大小由 maintain_model 的 prompt 控制
- MVP 规模（10-20 locations，100-200 observations）下 Model 远在上限内

当规模超出时（数十个 locations、数百条 observations），maintain_model 的 LLM 调用会面临输入过大的问题。此时可以：
- 只传最新 session 的 observations（而不是全量）
- 或引入 LLM 预压缩步骤，作为**性能优化而非架构必需**

---

## 六、停止条件

1. **Planner 调 mark_done 且 Verification PASS** — 侦察充分（Verification 由 mark_done 内部自动触发）
2. **安全网触发** — 总 tool call 数 / spawn_execution 次数达到上限
3. **Planner context window 满** — 强制结束

## 七、并发架构（预留）

### Phase 1：串行（MVP）

```
Planner tool-use 循环 → spawn_execution(Session 1) → spawn_execution(Session 2) → ...
```

### Phase 2：并行探索（未来）

```
Planner 一次 LLM response 返回多个 spawn_execution tool calls
  ├─ Session A (方向 1) ─┐
  ├─ Session B (方向 2) ─┤ 并发执行
  └─ Session C (方向 3) ─┘
  └─ 所有 Session 完成 → maintain_model 串行队列更新 Model → Planner 继续循环
```

并发安全由单例 Recording Agent 保证（所有 session 的 observations 写入经过同一个消费者）。Model 更新通过 maintain_model 串行队列保证一致性。

## 八、设计决策记录

### D1: 理解和判断全部 LLM 化，程序只做数据搬运

所有需要"理解"的工作（综合 observations、评估进展、决定方向、编写任务书）交给 LLM。程序化步骤仅负责从 DB 取数据并格式化成文本。

理由：程序化处理 observations 容易引入 schema mismatch 类 bug；LLM 直接读原始数据更可靠；符合"智能优先"硬约束。

### ~~D2: evaluate_and_decide 合并反思与决策~~（已废弃）

> 已被 Planner tool-use agent 架构取代。当前设计中 Planner 是持续 tool-use 循环，"提炼"由 maintain_model LLM 函数独立完成，"决策"由 Planner LLM 自主做。详见 `docs/Planner设计.md`。

### ~~D3: Briefing 包含原始 observations~~（已废弃）

> 已被 Planner tool-use agent 架构取代。当前 briefing 由 Planner 通过 `spawn_execution(briefing)` 参数自主生成，Planner 通过 `read_model()` 按需获取 Model 信息来编写 briefing。

### D4: Location 粒度由 agent 决定

Location 代表"站点上一个位置"，粒度不硬编码。agent 自己决定用 URL pattern（`/tag/{tag}`）还是更具体的形式。系统不强制归纳规则。符合"不预设网站类型"硬约束。

### D5: 数据膨胀由 Model 层自然压缩

Model 层本身就是 LLM 的压缩产物（maintain_model 全量重写）。Planner 只读 Model + 最新 session summary，不直接读历史 observations。不需要分层衰减格式化——分层衰减违反"智能优先"硬约束（程序化决定哪些 obs 该详细是 LLM 的工作）。

### D6: 并发预留，MVP 串行

录制 Agent 是各 session 的 Observations 唯一写入者，DB 事务保证并发安全。从串行到并行只需改 Planner 主循环。
