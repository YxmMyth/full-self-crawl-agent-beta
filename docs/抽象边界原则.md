# Agent vs Infrastructure — 抽象边界原则

> 起源:2026-04-28 ui8.net mission post-mortem。原本零碎的"该加什么工具/该藏什么细节"
> 决策反复,需要一个统一判定原则。本文是这条原则。

---

## 核心

**Agent 工作在语义层(意图 + 症状)。**
**Infrastructure 工作在机械层(实现细节)。**
**两层之间有抽象边界,信息穿越边界时翻译成对方层级能理解的形式。**

```
┌─────────────────────────────────────────┐
│  Agent layer                            │
│  - "我想知道这个站的数据结构"             │
│  - "fetch 返回了 401,意味着需要登录"     │
│  - "页面显示 'Sign In',我现在没登录"     │
│  - 决策、策略、规划、judgement            │
└───────────────────┬─────────────────────┘
                    │  symptom-level signals
                    │  ↑
                    │  ↓
                    │  semantic intent
┌───────────────────┴─────────────────────┐
│  Infrastructure layer                   │
│  - cookies.sqlite 在哪、TLS 怎么握手     │
│  - profile 分 4 层、sessionstore 关掉    │
│  - tab 怎么分配、popup 怎么关            │
│  - 进程生命周期、缓存、连接池             │
└─────────────────────────────────────────┘
```

---

## 判定问题(litmus test)

加新能力 / 暴露新信息 / 决定该藏还是该露,问自己 4 个问题:

1. **如果这个信息消失,agent 会做出不同决策吗?** 不会 → 藏
2. **这是"做什么"的事,还是"怎么实现"的事?** 怎么实现 → 藏
3. **这是单次诊断症状,还是常态机制状态?** 常态机制 → 藏
4. **agent 不知道这个事,任务能不能完成?** 能 → 藏

四个全 yes-towards-hide → 进 infra
四个全 yes-towards-expose → 进 agent 工具
混合 → 翻译:infra 处理机制,把症状暴露给 agent

---

## 翻译表(常见情形)

| 机械事件 | agent 看到的语义信号 |
|---------|-------------------|
| cookies 失效 / 过期 / 没登录 | HTTP 401 → `reason: auth_required` |
| 服务器 redirect 到登录页 | 返回 HTML + `reason: got_html_likely_login_redirect` |
| TLS 握手失败 / 网络断 | `reason: network_error` |
| Cloudflare 挑战 | 页面内容含挑战标记 / `reason: challenge_detected`(未来) |
| profile sessionstore 损坏 | infra sanitize 自愈,agent 不感知 |
| Camoufox 启动失败 | 系统层 raise,mission 中断,agent 不感知 |
| Tab 泄漏 / popup 自开 | infra 自动关闭,agent 不感知 |
| 浏览器引擎选择 | infra 决定,agent 看不到 "camoufox" 还是 "chromium" |
| 文件路径(profile / artifacts / cookies.sqlite 位置) | infra 内部用,agent 看不到 |

## 已经一致的应用

我们这一系列重构,事实上都是这条原则:

| 修复 | 收纳到 infra 的东西 |
|------|----------|
| sessionstore 关 + sanitize | profile Tier 3 文件管理 |
| 删 vanilla Chromium fallback | 浏览器引擎选择 |
| 单 tab 模型 + popup 拦截 | tab 生命周期 |
| fetch 工具(待开) | cookies / redirect / TLS / browser session |

## 反例(我们犯过的错)

- `browser_reset(browser_type='chromium' \| 'camoufox')` 把"引擎选择"暴露给 agent
  → 引擎是机械细节,agent 看不出 chromium vs camoufox 哪个反检测更好,选了等于乱选
  → **已修**:enum 砍到只剩 camoufox(实际上不让 agent 选)

- `browse(new_tab=, tab=)` 把"tab 操作"暴露给 agent
  → tab 不是 agent 决策依据(它本身只能看一个 tab),反而诱导 wasteful pattern
  → **已修**:browse 收成 `(url?, visual?)`

- 期望 agent 在 bash 里"记得带 cookie"
  → cookies 是 infra 状态,要 agent 处理 = 让 agent 懂 sqlite WAL / Mozilla cookies.txt 格式 / 转码
  → **修法**:加 fetch 工具,cookies 在 infra 内部处理(待开)

---

## 边界画错的两种典型危害

**画太低(太多细节暴露给 agent):**
- agent 把认知带宽花在机械操作上(管 tab、记 cookies、选引擎)
- 学到错误使用模式(开 22 个 tab、错路径下载)
- 对 prompt 教学敏感(每加一种机制就要教一遍)
- 工具表面膨胀

**画太高(太多动作藏进 infra):**
- agent 看不到失败原因(infra 把所有异常吞掉)
- 没法 react / 重试 / 换策略
- mission outcome 不可解释(为什么任务卡住?infra 不告诉它)
- "魔法"(magic),debug 困难

**正确位置:症状对 agent 透明,机制对 agent 隐藏。**
失败 → infra 翻译成具体 reason 给 agent → agent 在语义层 react。

---

## 长期意义

这条边界画清晰之后:

1. **加新工具的判断变简单** — 走判定问题清单,几乎自动得到答案
2. **agent prompt 简化** — 不需要教"如何带 cookie / 如何关 tab" 这种 mechanical 内容
3. **infra 改进自由度高** — 改 cookie 处理 / 引擎选择 / profile 分层,agent 完全无感
4. **测试更稳定** — agent 行为只跟症状和决策对,跟 infra 实现解耦
5. **未来并发 session / 多 agent 协作** — infra 层负责资源分配,agent 不感知

---

## 维护

新加任何 agent 工具或暴露任何信息,先过判定问题。文档里"翻译表"和"已经一致的应用"两节随实现更新。

如果出现"这个该 agent 决策但又涉及 mechanical detail" 的灰色情况,设计成 **agent 给意图 + infra 翻译成 mechanical 操作**(类似 fetch 接收 url + 自动处理 cookies),而不是 agent 直接操作 mechanical(类似 bash + curl 自己挑 cookie)。
