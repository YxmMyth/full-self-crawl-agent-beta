## 一、系统定位

### 输出的四层递进结构

```
第一层：Domain 理解
  这个站点是什么、数据怎么组织的

第二层：数据种类的存在形式/位置/数量
  站点上有哪些数据，通过什么路径可达，各路径的数据量
  同一实体在不同位置可能有不同表示和不同字段

第三层：具体需求对应的数据位置/数量
  用户的需求映射到哪些位置、预估规模

第四层：样本
  各位置实际看到的数据快照
  侦察过程中 extract() 自然产生，不需要独立的采样阶段
```

### 系统边界

系统到"理解 + 样本"为止。全量提取不在此次重构范围内。World Model 天然为全量提取提供完整输入（路径、方法、脚本），未来需要时直接从 World Model 读取。

### 核心原则

- **不建模，做田野笔记。** 不追求完整站点模型，目标是对需求相关区域的充分了解。数据理解从 observations 中涌现，不预设网站应该长什么样。
- **一种模式。** 探索和提取是同一任务的两个阶段，不是两个独立模式。侦察过程中的 extract 既是验证提取方法，也是产出样本。
- **探索服务于需求。** 用户有 requirement，探索不是漫无目的的，但探索过程中对"需求意味着什么"的理解本身也在进化。

## 二、探索方法论

### 三个层面，同一个循环

```
战略层：假设-验证
  防止 agent 迷失，给它方向感
  world model 中信息稀疏的 location 就是隐含的待验证假设

战术层：信息气味
  帮助 agent 在当前页面做出好的下一步决策
  体现为工具返回值中的结构化附加信息

执行层：自由探索
  agent 有 bash/浏览器/代码执行的完整权限
  不被硬编码限制，遇到意外情况能自己应对
```

运行时只有一个循环——Planner 的 tool-use 循环：

```
Planner.run(domain, requirement):
    while not stopped:
        LLM 决定下一个 tool call
        → spawn_execution / spawn_research / read_model / think / mark_done
```

详见 `docs/Planner设计.md`。

## 三、ReconPlanner

### 定位

两层架构的战略决策层。以 **tool-use agent** 形式运行（持续 LLM 循环 + 5 个工具）。

详细设计见 `docs/Planner设计.md`。

### 职责

```
做什么：
  ✅ 通过 spawn_execution / spawn_research 派发子 agent
  ✅ 通过 read_model 获取当前认知状态
  ✅ 通过 mark_done 标记完成（内部自动触发 Verification）
  ✅ 通过 spawn_execution 的 briefing 参数生成任务书
  ✅ 管理总预算（安全网参数）

不做什么：
  ❌ 不直接操作浏览器
  ❌ 不直接维护 observations
  ❌ 不直接更新 Model（由 maintain_model LLM 函数自动触发）
  ❌ 不决定 agent 在 session 内部去哪个 URL
  ❌ 不微管理探索顺序
```

### Planner tool-use 循环 + Session 内部

```
Planner（tool-use agent）
  Initial: "Domain: {domain}, Requirement: {requirement}"
  ↓
  LLM 决定 tool call → spawn_execution / spawn_research / read_model / think / mark_done
  ↓
  [spawn_execution(briefing)] 的完整 pipeline：
    ├── 启动 Session
    │     ├── 执行 Agent（tool-use 循环，12 个工具）
    │     └── 单例 Recording Agent 持续维护 Observations（CRUD）
    │          transcript 增量由 Python 代码推给 Recording Agent
    ├── Session 停止（自然结束 / context 满 / 连续失败）
    ├── transcript 程序化保存
    ├── maintain_model（Python 代码自动触发的 LLM 函数调用）
    │     读 current Model + new observations → 更新 Model → 写回 DB
    └── 返回 { summary, model_diff, new_obs_count } → push 到 Planner context
  ↓
  [spawn_research(topic, questions)] 的 pipeline：
    ├── 启动 Research Subagent（4 个工具：web_search/web_fetch/bash/think）
    └── 返回 { key_findings, report_path } → push 到 Planner context
  ↓
  [read_model()] → pull 当前完整 Semantic + Procedural Model
  ↓
  [mark_done(reason)] 的 pipeline：
    ├── if VERIFICATION_SUBAGENT_ENABLED:
    │     自动运行 Verification Subagent（3 个工具：read_world_model/bash/think）
    │     PASS → 返回 { status: "DONE" }
    │     FAIL/PARTIAL → 返回 { status: "blocked", gaps: ... }
    └── else: 返回 { status: "DONE" }
  ↓
  Planner 看到 tool_result → LLM 决定下一步 → 循环继续
```

**关键设计：**
- ~~evaluate_and_decide + generate_briefing 两次固定 LLM 调用~~ → Planner 自主 tool-use 循环
- Model 更新不由 Planner 操作——spawn_execution 的 Python 实现里自动调 maintain_model
- Verification 不由 Planner 主动 spawn——mark_done 的 Python 实现里自动触发
- Research 触发完全由 Planner LLM 自主判断（不依赖 signal 机制）
- 首次 briefing 无特殊分支——Planner 第一次 tool call 就是 spawn_execution

详见 `docs/Planner设计.md`、`docs/系统架构与信息流.md`。

### Briefing

Planner 通过 `spawn_execution(briefing)` 的参数生成任务书。Briefing 是 Planner 从 requirement 和 Model 派生的自然语言指令，暴露多少 requirement 由 Planner 根据当前 L1-L4 阶段判断（见 `docs/Planner设计.md §四`）。

### confidence

不需要独立字段。Semantic Model 自然表达不确定性，Planner LLM 从 Model 的信息密度中自己判断。

### 整体任务停止条件

```
1. Planner 调 mark_done → Verification PASS → 真正结束
2. 安全网触发：总 tool call 数上限 / spawn_execution 次数上限
3. Context window 满
```

### Requirement 是顶级不变量

系统输入只有 domain + requirement。Requirement 贯穿整个 run，但**不传给 Execution Agent**（只在 Planner / Verification / Research 层可见）。Planner 根据 L1-L4 阶段决定 briefing 中暴露多少 requirement。详见 `docs/Planner设计.md §四`。

## 四、Agent 自主性原则

- Agent 有浏览器、bash、代码执行的完整权限
- Agent 自己决定去哪、看什么、提什么
- 不硬编码控制流
- 每个站点不同，路径不可预测，硬编码必然脆弱

## 五、Agent 行为设计

### Agent 的思维原则（写入 system prompt）

```
1. 先观察再行动
2. 对数字保持怀疑，多源交叉验证
3. 发现就记录（录制 Agent 会捕捉你的推理和发现），不要攒着
4. 探索多条路径
5. 注意关系，发现联系就明确表述（录制 Agent 会捕捉）
```

### System Prompt 分层

```
system prompt：思维原则（短，稳定）→ 教 agent 怎么想
briefing（user message）：当前理解和方向（动态）→ 告诉 agent 想什么
```

### Session 停止机制

```
第一层：LLM 不再调工具 → 自然结束
第二层：Context window 满了 → 强制结束
第三层：连续失败检测 → 强制结束
```

不需要特殊停止标记。安全中断保障：录制 Agent 实时维护 Observations。

## 六、工具集设计

> 详细设计见 `docs/工具重新设计共识.md`

### 工具集（按能力层）

```
浏览器感知（三个不可互替的能力层）：
  browse(url?, new_tab?, tab?)       — 页面内容快照 + 多标签页导航
  read_network(filter?)              — 网络层信息（请求/响应/cookies）
  browser_eval(script, save_as?)     — 浏览器内 JS 执行（探测/提取/检查）

浏览器管理：
  browser_reset(proxy?, browser_type?, headed?)
                                     — 重启浏览器到新配置（每 agent 独占）

页面交互（5 个，按参数模式拆分）：
  click(target)                      — 点击元素（自动识别 <select>）
  input(target, value)               — 输入/选择（自动检测 autocomplete + 值验证）
  press_key(key, target?)            — 按键（支持组合键）
  scroll(direction?, amount?, target?) — 滚动（支持水平 + 容器内）
  go_back()                          — 浏览器后退

系统执行：
  bash(command)                      — 浏览器外代码执行

认知辅助：
  think(thought)                     — 推理
  read_world_model(location?)        — 查询 World Model（无参数=完整Models，有location=该location的Observations）
```

### 设计原则

- 一工具一能力，不做 enum 多路复用
- 无硬编码检测逻辑（遵守硬约束 #6"智能优先"）
- 浏览器感知按不可互替的 Playwright 能力层拆分
- browser_eval + bash 是一对：浏览器内执行 vs 浏览器外执行

### 浏览器环境与策略

**默认配置：Camoufox headless（能力最优直接上）**

- 本地开发：Camoufox headed（可视，方便调试）
- 生产/并发：Camoufox headless（C++ 层反检测在 headless 下同样生效，省资源）
- 每个 agent 独占一个浏览器实例（1 agent = 1 site = 1 browser）

**browser_reset 支持的浏览器类型：**
- `camoufox`（默认，Firefox 反检测，C++ 层指纹伪造）
- `chromium`（兼容性 fallback，Firefox 不兼容时使用）
- `patchright`（Chromium + 反检测，以后按需加）

**问题应对策略（agent 自主判断，不硬编码升级链）：**

| 问题 | 应对 | 用什么工具 |
|------|------|-----------|
| Firefox 不兼容 | 换 Chromium | `browser_reset(browser_type="chromium")` |
| TLS/JA3 指纹被识别 | HTTP 直接请求 | `bash` + curl_cffi（模拟浏览器 TLS） |
| IP/ASN 被拦 | 加代理 | `browser_reset(proxy="socks5://...")` |
| CAPTCHA | 调 solver API | `bash` 调 CapSolver 等服务 |
| 极端保护（DataDome 等，~2.8% 站点） | 未来集成云浏览器服务 | MVP 不覆盖 |

**关键数据（来自 2025 行业调研）：**
- 61.2% 网站完全无反爬保护，36% 部分保护，仅 2.8% 完整保护
- Camoufox headless 的 C++ 反检测覆盖 canvas/WebGL/字体/音频/navigator，headed 与 headless 反检测效果一致
- 住宅代理成功率 85-95% vs 数据中心 20-40%（需要时才加，不默认）
- CAPTCHA solver ~$0.02/次（bash 可调用）

**浏览器基础设施层（src/browser/）：**

Camoufox 只提供浏览器进程，工具层需要的共享基础设施由 `src/browser/` 提供：

```
src/browser/
├── manager.py          # 连接管理（直接启动 / WS / CDP / Chromium fallback）
├── page_repr.py        # 页面表示（HTML → Markdown+HTML 混合格式）
├── element_index.py    # 元素索引系统（编号 ↔ selector 映射）
├── network_capture.py  # 网络请求捕获（page.on('response')，不用 page.route()）
├── dom_settle.py       # DOM 稳定等待 + SPA 空壳检测
└── context.py          # 工具共享的浏览器上下文（page, selector_map, captures）
```

注：网络捕获必须用被动监听 `page.on('response')`，不能用 `page.route()`——后者会触发 PerimeterX 等反检测系统。

## 七、记忆与存储系统

> 数据模型详见 `docs/WorldModel设计.md`

### 7.1 三类存储

```
1. 内存（session 内工作记忆）
   └── 执行 Agent 的 message array
       包含：briefing + 工具调用 + 返回值 + agent 推理文本
       生命周期：session 开始到结束
       agent 的 context 就是它的工作记忆

2. 数据库（跨 session 持久化，World Model）
   ├── locations      — 访问过的 URL/位置
   ├── observations   — 结构化观察（录制 Agent 写入）
   ├── models         — Semantic Model + Procedural Model（maintain_model 更新）
   └── sessions       — session 元数据

3. 文件系统（跨 session 持久化，per-domain）
   └── artifacts/{domain}/
       ├── samples/        — 提取的数据样本（browser_eval save_as）
       ├── scripts/        — 可复用脚本（精准记忆）
       ├── workspace/      — 临时文件、bash 大输出落盘
       └── transcripts/    — session 完整记录
```

### 7.2 数据流转

```
① 执行 Agent + 录制 Agent 并行运行
   执行 Agent：message array 自然积累（工具调用 + 返回值 + agent 推理文本）
   录制 Agent：每轮 tool call 后 async 触发，读当前 message array
              → 维护 Observations（创建/更新/合并/删除，执行 Agent 可通过 read_world_model 查回）
   browser_eval save_as → artifacts/{domain}/samples/
   bash 大输出自动落盘 → artifacts/{domain}/workspace/（返回预览 + 文件路径）

② Session 结束
   完整 message array 保存到 artifacts/{domain}/transcripts/（基础设施，程序化）

③ maintain_model（Python 代码自动触发的 LLM 函数，非 agent）
   输入：current Model + 本 session new observations
   输出：更新 Semantic + Procedural Model → 写回 DB models 表
   同时生成 session summary + model_diff

④ 返回给 Planner（push）
   Planner 的 spawn_execution tool_result 包含 summary + model_diff
   Planner 需要完整 Model 时调 read_model（pull）
```

### 7.3 两种记忆

| | 语义记忆 | 精准记忆 |
|---|---------|---------|
| **内容** | 站点结构理解、关系、策略判断 | 可执行脚本、具体数据、API 参数 |
| **存在哪** | Semantic/Procedural Model（DB，有界） | 文件系统（scripts/ samples/，无界） |
| **引用方式** | Model 文档里写文件路径 | 路径即地址 |
| **例子** | "tag 页有 __NEXT_DATA__，含 pen 列表" | `scripts/extract_tag_pens.js` 完整代码 |
| **谁写** | maintain_model LLM 函数（Python 代码自动触发） | 执行 Agent（browser_eval save_as / bash） |

**Model 是索引，文件是内容。** Procedural Model 记"tag 页提取方法：__NEXT_DATA__，脚本见 scripts/extract_tag_pens.js"，文件存完整可执行代码。

### 7.4 录制 Agent（单例 + Producer-Consumer）

**架构：与执行 Agent 同构——持久对话 + tool-use 循环，只是输入和工具不同。**

**单例设计：** 所有 Session 共享同一个录制 Agent 实例（Producer-Consumer 模式）。

```
Session A 的 transcript 增量 ─┐
Session B 的 transcript 增量 ─┼→ [单例 Recording Agent] ─→ observations DB
Session C 的 transcript 增量 ─┘

单例 Recording Agent（持久对话）
  ├── 感知：各 session 的 transcript 增量（Python 代码推送，标记 session_id）
  ├── 感知：当前 observations（通过 read_observations 工具）
  └── 行动：维护 observations（通过 create/edit/delete 工具）
        │
        └──→ 写入 DB → 执行 Agent 调 read_world_model 查回
```

**为什么单例：**
- 多 session 并发时，多个录制 Agent 写 observations 会重叠/冲突
- 串行下跨 session 的 location 重叠也需要全局视角去重/合并
- 单例消费者保证没有写冲突
- 批处理/过滤优化：只有产生新知识的 tool call（browse/bash/browser_eval）才触发处理

**录制 Agent 的 LLM 对话结构：**

```
System: "你是录制 Agent，观察执行过程，维护结构化 observations..."

User: "执行更新：[step 1-3 的 transcript 增量]"
Assistant: → read_observations(location="/tag/*")
          → create_observation(location="/tag/*", raw={...})

User: "执行更新：[step 4-6 的 transcript 增量]"
Assistant: → read_observations(location="/tag/*")
          → edit_observation(id=5, raw={...updated...})

User: "执行更新：[step 7-9 的 transcript 增量]"
Assistant: "这几步没有新发现值得记录"（no tool call）
```

**两个 Agent 的对比：**

| | 执行 Agent | 录制 Agent |
|---|-----------|-----------|
| System prompt | 侦察方法论 + 思维原则 | 观察纪律 + 记录策略 |
| User input | Briefing（Planner 生成） | 执行 transcript 增量（程序注入） |
| Tools | 12 个（浏览器 + bash + 认知） | 4 个（read/create/edit/delete observations） |
| 输出 | 浏览器操作、数据提取 | 结构化 observations 写入 DB |
| Context | 自己的工具调用历史 | 执行 transcript + 自己的工具调用历史 |

**录制 Agent 的工具集（参考 Claude Code 的 Read/Edit/Write 模式）：**
- `read_observations(location?)` — 查看当前 observations（先看再改）
- `create_observation(location, raw)` — 创建新 observation
- `edit_observation(id, raw)` — 更新已有 observation
- `delete_observation(id)` — 删除冗余/已合并的 observation

**录制 Agent 的输入不只是工具返回值——agent 的推理文本也是观察。** 执行 Agent 在比较、推断、发现关系时的思考过程，同样是有价值的观察数据。

**设计理由：**
- Agent 不负责记录——它的 context 就是工作记忆，记录会分散注意力
- 录制 Agent 与执行 Agent **实时并行**——程序定期注入 transcript 增量，不阻塞执行
- **mid-session 知识回忆**——执行 Agent 后期可通过 read_world_model 查回早期发现（解决 context 截断导致的信息丢失）
- **Observations 可维护**——先 read 再改，跟 Claude Code 编辑代码一样的模式。避免数量爆炸，保持知识层精炼
- **Transcript 是不可变审计追踪**——Observations 的任何修改都可追溯
- 录制是智能的——LLM 判断什么值得记、什么粒度、什么跟之前的发现有关联、旧 observation 是否需要更新
- 可用更轻模型——任务相对简单（观察 + 维护结构化知识）

### 7.4b Research Subagent（按需 spawn）

**定位：** Planner 的辅助调研能力。由 Planner 通过 `spawn_research(topic, questions)` 工具自主 spawn。

**触发时机：** 完全由 Planner LLM 自主判断——可以在任何时候 spawn（初始调研、中途遇到未知 tech stack、验证失败后需要查文档等）。没有硬编码的 signal 机制。

**架构：与执行 Agent 同构（tool-use 循环）**

```
Planner tool-use 循环
  └─ spawn_research(topic, questions) → Research Subagent
        │
        ▼
Research Subagent（独立 tool-use 循环）
  System prompt：调研纪律 + 输出结构
  User input：调研任务描述（topic + questions）
  Tools：
    - web_search(query, domain?)  — 搜索引擎查询
    - web_fetch(url)              — 轻量 HTTP 获取（不走浏览器）
    - bash(command)               — 兜底能力（curl / gh api 等）
    - think(thought)              — 无副作用推理
  输出：调研报告 markdown → artifacts/{domain}/research/{topic}.md
        ↓
返回 { key_findings, report_path } → push 到 Planner context
```

**与执行 Agent 的对比：**

| | 执行 Agent | Research Subagent |
|---|-----------|-----------------|
| 感知对象 | 目标站点 | 互联网 |
| 工具数 | 12 | 4 |
| 输出 | Observations → WM | 调研报告 → 文件 |
| 生命周期 | session 级（长） | 任务级（短，跑完即结束） |

**设计理由：**
- Research 是辅助能力，不应耦合到执行 Agent 的工具集（避免分散注意力）
- 触发完全由 Planner LLM 自主判断，不依赖硬编码的 signal 机制
- 调研报告存文件，跟 scripts/ samples/ 一样是精准记忆的载体
- 搜索后端可切换（MVP: DuckDuckGo → 生产: Brave/Tavily）

### 7.4c Verification Subagent（DONE 守门人，feature-gated 可拆卸）

**定位：** DONE 守门人，反 satisficing 机制。**由 `mark_done` 工具的 Python 实现内部自动触发**——Planner 不直接 spawn，不需要知道 verification 存在。通过 feature flag 可完全禁用。

**触发机制：**

```
Planner 调 mark_done(reason)
  ↓ Python 代码实现
  if VERIFICATION_SUBAGENT_ENABLED == false:
    return { status: "DONE" }
  else:
    自动运行 Verification Subagent
        │
        ▼
    Verification Subagent（独立 tool-use 循环）
      System prompt：反 satisficing 纪律（借鉴 Claude Code Verification Agent）
      User input：
        - 当前 WM 快照（Semantic Model + Procedural Model）
        - Requirement 原文
        - mark_done 的 reason
      Tools：
        - read_world_model(location?)  — 读 WM（全局 / 局部）
        - bash(command)                — 执行验证（curl API / 跑脚本 / 读 samples）
        - think(thought)               — 推理
      输出：
        - 报告 markdown → artifacts/{domain}/verification/round_N.md
        - 最后一行：VERDICT: PASS / FAIL / PARTIAL
        │
        ▼
  if VERDICT == PASS: return { status: "DONE" }
  if VERDICT == FAIL/PARTIAL:
    return { status: "blocked", verdict, gaps }
    → Planner 看到 blocked + gaps → 继续 tool-use 循环解决 gaps
```

**为什么由 mark_done 内部触发而不是 Planner 主动 spawn：**
- LLM 容易偷懒跳过 verification（system prompt 约束不可靠）
- 程序触发不依赖 LLM 良心
- Planner 的 tool-use 循环里只知道"mark_done 可能失败"，不知道 verification 存在

**反 satisficing 的核心机制（system prompt 的四个要素）：**

1. **命名 rationalization**——prompt 里直接列出 agent 会想的偷懒借口，强制反向动作：
   - "Semantic Model 看起来很完整" → 读文档不是验证，跑命令
   - "evaluate_and_decide 已经判断 DONE 了" → evaluate 是 LLM，独立验证
   - "这个 location 大概率是对的" → "大概率"不是验证
   - "Procedural Model 写了提取方法" → 读方法不算验证，跑一遍
2. **"读 WM 不是验证"**——禁止纯语言层面的检查，必须有执行动作
3. **强制 adversarial probe**——PASS 之前必须至少有一个主动找漏洞的动作（矛盾检查 / 覆盖检查 / 方法验证 / 路径对齐）
4. **严格输出格式 + 三选一裁决**——PASS/FAIL/PARTIAL 必须是最后一行，格式可被程序解析

**与 Research Subagent / 执行 Agent 的对比：**

| | 执行 Agent | Research Subagent | Verification Subagent |
|---|-----------|-----------------|---------------------|
| 感知对象 | 目标站点 | 互联网 | WM + 执行侧验证 |
| 工具数 | 12 | 4 | 3 |
| 输出 | Observations → WM | 调研报告 → 文件 | 验证报告 + VERDICT → 文件 |
| 权限 | 读写 WM | 不读 WM | **只读 WM，不写** |
| 触发时机 | 每个 session | Planner 自主 spawn | mark_done 内部自动触发（feature 启用时） |
| 是否必需 | 是 | 否（MVP 可选） | 否（feature-gated） |

**工具选择理由：**
- **只 3 个工具**——MVP 最小化，比 Research Subagent 还精简
- **不给浏览器工具**——浏览器启动复杂，bash 可以通过 curl / curl_cffi 验证 API、读 samples、跑 scripts
- **严格只读 WM**——不能 create/edit/delete observations，跟 Claude Code Verification Agent "cannot modify project files" 一致
- 后续扩展：browser 相关验证可以未来加 browser_eval（只读模式）

**设计理由：**
- **只在 mark_done 时触发**——不是每轮都跑，避免成本爆炸
- **blocked 返回值驱动 Planner 继续**——Planner 看到 gaps 后自然处理（spawn_execution 解决 gaps）
- **Feature flag 完全拆卸**——禁用后 mark_done 直接返回 DONE。代码保留但不执行
- **借鉴 Claude Code 生产级 prompt**——Verification Agent 是 Claude Code 内置的反 satisficing 机制，prompt 纪律有生产验证
- **程序触发 >> system prompt 约束**——不依赖 LLM 良心，架构层面保证

### 7.5 Session Transcript（不可变记录层）

**定位：** 三层数据架构的最底层——不可变的原始记录。Observations 的任何修改都可追溯到 transcript。

**保存方式：** 基础设施层程序化保存，零 LLM 成本。每条消息（role + content + tool_calls + timestamp）序列化为 JSONL。

**三重用途：**

| 用途 | 消费者 |
|------|--------|
| 不可变审计追踪（Observations 是可维护的，transcript 是不可变的原始记录） | 全系统 |
| 调试/定位问题 | 开发者 |
| 行为分析/优化 | 开发者 |

**保存位置：** `artifacts/{domain}/transcripts/{session-id}.jsonl`

### 7.6 Read（信息获取模型）

```
Planner 层（Push + Pull）：
  Push：spawn_execution / spawn_research 的 tool_result 自动返回摘要
  Pull：Planner 调 read_model() → 完整 Semantic + Procedural Model

Session 层（Pull）：
  主动：执行 Agent 调 read_world_model（两层检索）
    无参数 → 完整 Semantic + Procedural Model（全局概览）
    指定 location → 该 location 的 Observations（局部证据）
  精准回忆：agent 用 bash 读取 scripts/ 和 samples/ 中的文件
```

## 八、跨 Run 复用

World Model 通过 DB 自然持久化。同一域名再次运行时：

```
1. 从 DB 加载旧的 locations / observations / models
2. Planner 通过 read_model 看到旧 Model（带时间戳），自己判断旧知识的可信度
3. 不硬编码过期策略——LLM 判断"3 天前的站点结构大概率没变"
   vs "半年前的提取脚本可能需要重新验证"
4. Agent 可以选择验证旧知识或探索新方向
```

## 九、设计约束

1. **不预设网站类型。**
2. **不预设数据 schema。**
3. **不预设关系类型枚举。**
4. **Transcript 只追加不修改。** Observations 由录制 Agent 维护（CRUD），Transcript 是不可变的审计追踪。
5. **三层数据架构。** Transcript（不可变）→ Observations（录制 Agent 维护）→ Models（maintain_model LLM 函数全量重写）。详见 docs/WorldModel设计.md。
6. **智能优先。** 信息的组织、筛选、呈现交给 LLM，不硬编码规则。
