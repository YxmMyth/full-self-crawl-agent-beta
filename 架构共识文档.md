## 一、系统定位

### 输出的四层递进结构

```
第一层：Domain 理解
  这个站点是什么、数据怎么组织的

第二层：数据种类的存在形式/位置/数量
  站点上有哪些数据，通过什么路径可达，各路径的数据量
  同一实体在不同位置可能有不同表示和不同字段

第三层：具体需求对应的数据位置/数量
  用户的需求映射到哪些位置、预估规模

第四层：样本
  各位置实际看到的数据快照
  侦察过程中 extract() 自然产生，不需要独立的采样阶段
```

### 系统边界

系统到"理解 + 样本"为止。全量提取不在此次重构范围内。World Model 天然为全量提取提供完整输入（路径、方法、脚本），未来需要时直接从 World Model 读取。

### 核心原则

- **不建模，做田野笔记。** 不追求完整站点模型，目标是对需求相关区域的充分了解。数据理解从 observations 中涌现，不预设网站应该长什么样。
- **一种模式。** 探索和提取是同一任务的两个阶段，不是两个独立模式。侦察过程中的 extract 既是验证提取方法，也是产出样本。
- **探索服务于需求。** 用户有 requirement，探索不是漫无目的的，但探索过程中对"需求意味着什么"的理解本身也在进化。

## 二、探索方法论

### 三个层面，同一个循环

```
战略层：假设-验证
  防止 agent 迷失，给它方向感
  world model 中信息稀疏的 location 就是隐含的待验证假设

战术层：信息气味
  帮助 agent 在当前页面做出好的下一步决策
  体现为工具返回值中的结构化附加信息

执行层：自由探索
  agent 有 bash/浏览器/代码执行的完整权限
  不被硬编码限制，遇到意外情况能自己应对
```

运行时只有一个循环：

```
while not done:
    evaluate_and_decide → generate_briefing → execute session
```

## 三、ReconPlanner

### 定位

Planner-Executor 两层架构中的 Planner 层。

### 职责

```
做什么：
  ✅ 管理 SiteWorldModel 的生命周期（含跨 run 持久化）
  ✅ session 后反思（reflect）：从新 observations 中提炼高层 insight
  ✅ session 间战略判断（decide_next）：继续/停止/方向调整
  ✅ 给每个 session 生成任务简报（generate_briefing）
  ✅ 管理总预算
  ✅ 编译最终报告

不做什么：
  ❌ 不决定 agent 在 session 内部去哪个 URL
  ❌ 不维护独立的假设列表
  ❌ 不微管理探索顺序
  ❌ 不管理浏览器实例
```

### Session 间流程

```
Session N 结束
  ↓
evaluate_and_decide(wm):
  读取本轮 session 的新 observations + 完整 World Model
  LLM 同时完成：
    1. 更新 Semantic Model（站点结构理解、数据关系）
    2. 更新 Procedural Model（已验证的提取方法）
    3. 判断 DONE / CONTINUE + 方向
  Model 更新是决策的副产品，不是独立步骤
  ↓
如果继续 → generate_briefing(wm, direction):
  LLM 从 World Model 生成给 agent 的任务简报
  ↓
Session N+1 开始
```

详见 docs/系统架构与信息流.md。

### generate_briefing()

LLM 驱动的信息整合，不硬编码模板。因为 World Model 包含了 Semantic/Procedural Model，briefing 质量自然高。

### confidence

不需要独立字段。Semantic Model 自然表达不确定性，evaluate_and_decide 的 LLM 从信息密度中自己判断。

### 整体任务停止条件

```
1. evaluate_and_decide 判断信息足够 → DONE
2. 总预算耗尽
3. 连续 N 个 session 没有新发现
```

## 四、Agent 自主性原则

- Agent 有浏览器、bash、代码执行的完整权限
- Agent 自己决定去哪、看什么、提什么
- 不硬编码控制流
- 每个站点不同，路径不可预测，硬编码必然脆弱

## 五、Agent 行为设计

### Agent 的思维原则（写入 system prompt）

```
1. 先观察再行动
2. 对数字保持怀疑，多源交叉验证
3. 发现就记录（note_insight），不要攒着
4. 探索多条路径
5. 注意关系，发现联系就 note_insight 记录
```

### System Prompt 分层

```
system prompt：思维原则（短，稳定）→ 教 agent 怎么想
briefing（user message）：当前理解和方向（动态）→ 告诉 agent 想什么
```

### Session 停止机制

```
第一层：LLM 不再调工具 → 自然结束
第二层：步数预算耗尽 → 强制结束
第三层：连续失败检测 → 强制结束
```

不需要特殊停止标记。安全中断保障：note_insight 实时写入 World Model。

## 六、工具集设计

> 详细设计见 `docs/工具重新设计共识.md`

### 工具集（按能力层）

```
浏览器感知（三个不可互替的能力层）：
  browse(url?)                       — 页面内容快照（Markdown+HTML 混合）
  read_network(filter?)              — 网络层信息（请求/响应/cookies）
  browser_eval(script, save_as?)     — 浏览器内 JS 执行（探测/提取/检查）

页面交互：
  interact(action, target, value?)   — 统一交互（target 为元素编号）

系统执行：
  bash(command)                      — 浏览器外代码执行

认知辅助：
  think(thought)                     — 推理
  note_insight(content, location?)   — 记录洞察（含关系描述）
  read_world_model(section?)         — 查询 World Model
```

### 设计原则

- 一工具一能力，不做 enum 多路复用
- 无硬编码检测逻辑（遵守硬约束 #6"智能优先"）
- 浏览器感知按不可互替的 Playwright 能力层拆分
- browser_eval + bash 是一对：浏览器内执行 vs 浏览器外执行

### 浏览器环境

默认 camoufox。Agent 可绕过用 bash。不预设访问障碍枚举。浏览器实例管理是 agent 自己的事。

## 七、知识记录架构

> 详细设计见 `docs/WorldModel设计.md`

### 三层记忆架构

```
Episodic（事件记忆）：
  Location     — 站点上的位置/路径
  Observation  — 在位置上看到的/理解的东西（自由格式 JSONB，只追加）
  sessions     — 运行历史记录

Semantic Model（语义模型）：
  LLM 维护的有界文档（~3000 chars）
  站点结构理解、数据关系、关键发现
  由 evaluate_and_decide 作为决策副产品更新

Procedural Model（程序模型）：
  LLM 维护的有界文档（~2000 chars）
  已验证的提取方法、有效路径
  由 evaluate_and_decide 作为决策副产品更新
```

### Write

```
程序自动：session 结束 → sessions 记录
Agent 主动：note_insight → Observation
ReconPlanner：evaluate_and_decide → 更新 Semantic Model + Procedural Model
```

WM 自动记账（browse/browser_eval 是否自动写 Observation）待讨论，可能剥离到基础设施层。

### Read

```
被动：generate_briefing 的 LLM 读完整 World Model → 生成简报注入 agent context
主动：agent 调 read_world_model 查原始细节
```

### 存储

```
PostgreSQL：locations / observations / models / sessions
内存缓存：SiteWorldModel 对象 = DB 镜像
LLM Context：briefing + session 内 tool call 历史
```

## 八、跨 Run 复用

World Model 通过 DB 自然持久化。同一域名再次运行时：

```
1. 从 DB 加载旧的 locations / observations / models
2. generate_briefing 的 LLM 看到时间戳，自己判断旧知识的可信度
3. 不硬编码过期策略——LLM 判断"3 天前的站点结构大概率没变"
   vs "半年前的提取脚本可能需要重新验证"
4. Agent 可以选择验证旧知识或探索新方向
```

## 九、设计约束

1. **不预设网站类型。**
2. **不预设数据 schema。**
3. **不预设关系类型枚举。**
4. **Observation 只追加不修改。**
5. **三层记忆架构。** Episodic（Observations）+ Semantic Model + Procedural Model。详见 docs/WorldModel设计.md。
6. **智能优先。** 信息的组织、筛选、呈现交给 LLM，不硬编码规则。
